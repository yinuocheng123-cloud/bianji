# 版本记录：v0.6-recent-failure-details

## 1. 本轮目标

给三步工作台补上“最近失败明细”，让异常面板从“知道失败数量”升级到“知道失败发生在哪条队列、哪条内容、报了什么错”。

这轮目标属于：

- 功能补充
- 可维护性提升
- 异常定位效率优化

优先做这件事的原因是，上一轮虽然已经可以在首页看到失败任务数量并发起重试，但如果值班人员看不到具体失败信息，仍然要再跳去日志里二次排查，工作台的异常闭环就还不够完整。

## 2. 背景与上下文

本轮承接：

- `custom/notes/v0.5-review-comment-modal.md`

在这之前，系统已经具备：

- 三步工作台主入口
- 队列健康状态
- 失败任务重试
- Worker 失败写日志

因此这轮不需要再动底层执行链路，只需要把已经存在的失败信息更高效地带回首页即可。

## 3. 问题分析

表面现象：

- 首页知道失败任务有多少
- 但不知道最近失败的是哪条队列
- 也不知道错误原因大致是什么

核心矛盾：

- 失败信息已经存在于日志里，但还没有被工作台直接消费

主要原因：

1. 之前优先补的是失败数量和重试能力
2. 最近失败明细没有被纳入首页服务端查询
3. 首页异常面板缺少“从统计走向排查”的最后一步

## 4. 候选方案比较

### 方案 A：扩展控制状态接口，把失败明细也塞进去

核心思路：

- 修改 `/api/control/status`
- 把最近失败列表一起返回给客户端

优点：

- 前端数据入口统一

缺点：

- 控制接口会继续膨胀
- 首页已有大量服务端查询，再走客户端刷新链路不一定最简

放弃原因：

- 这轮只是补首页展示，没有必要继续扩大控制接口职责

### 方案 B：首页服务端直接读取最近失败日志

核心思路：

- 复用 Worker 已经写入的 `queue:failed` 日志
- 在首页服务端查询时直接取最近失败明细
- 渲染成工作台下方的独立卡片

优点：

- 改动最小
- 不增加新的接口复杂度
- 完全复用现有日志链路

缺点：

- 数据只在首页服务端渲染，不走控制状态接口

最终选择原因：

- 这最符合当前目标，也最符合“最小改动”原则

## 5. 最终决策

本轮最终采用方案 B，具体决策如下：

1. 首页服务端新增最近失败日志查询
2. 解析 `queue:failed` 日志中的：
   - queue
   - message
   - contentItemId
   - targetId
3. 在工作台底部新增“最近失败明细”卡片
4. 保留“查看完整日志”入口，便于继续深挖

## 6. 具体实现

本轮修改文件如下：

- `src/app/(dashboard)/dashboard/page.tsx`
- `AGENTS.md`

新增正式记录：

- `custom/notes/v0.6-recent-failure-details.md`

### `src/app/(dashboard)/dashboard/page.tsx`

主要改动：

- 在原有 `Promise.all` 查询中新增最近失败日志读取
- 只取 `action = queue:failed`
- 将 `detail` 中的 `queue`、`message`、`data.contentItemId` 提取成首页可直接展示的结构
- 在页面底部新增“最近失败明细”卡片

展示内容包括：

- 队列名
- 模块名
- 时间
- 任务 ID
- 内容 ID
- 报错信息

此外补了一个 [`/logs`](/D:/ceshi/bianji/src/app/api/logs/route.ts) 入口按钮，方便继续查看完整日志。

## 7. 本轮优点

本轮带来的主要好处：

- 工作台异常面板不再只有数字
- 值班编辑可以直接判断失败大致发生在哪条链路
- 排查路径更短，不需要第一时间跳日志页
- 完全复用现有日志结构，没有额外引入新数据源

得到改善的质量属性：

- 可观察性
- 排障效率
- 可维护性

## 8. 本轮缺点与代价

本轮仍有这些限制：

1. 最近失败明细目前只展示最近几条，不是完整失败列表
2. 错误信息仍然是原始 message，没有做更高级的分组或归类
3. 还不能直接从失败明细跳到对应内容详情
4. 工作台仍然没有失败趋势或按队列聚合分析
5. 真实 OpenAI 联调仍未开始

这些代价当前可以接受，因为本轮重点是先把“最近失败在哪”补齐，而不是一次性做成完整运维面板。

## 10. 本轮结论

本轮最重要的结论是：

三步工作台现在已经从“知道失败数量”进一步升级到“知道最近失败在哪”，异常面板真正开始具备排查价值。

被证实的判断：

- 复用现有 Worker 日志链路就足以把失败明细快速挂回首页
- 不需要继续增加接口复杂度，也能明显提升工作台实用性

后续应默认继承的经验：

- 异常相关增强优先考虑复用现有日志，而不是先扩接口
- 工作台首页适合承载“最近失败摘要”，完整明细仍交给日志页

## 11. 下一轮建议

下一步最值得做的 1 到 3 件事：

1. 给失败明细增加直达内容或草稿的入口
   - 收益最高
   - 能从“知道失败了什么”推进到“马上去处理”

2. 给失败信息做简单分类
   - 风险最低
   - 能帮助编辑快速区分抓取失败、抽取失败还是草稿生成失败

3. 配置真实 `OPENAI_API_KEY` 并做 AI 抽取、草稿生成联调
   - 仍然是生产能力最关键的一步
   - 需要外部配置配合
