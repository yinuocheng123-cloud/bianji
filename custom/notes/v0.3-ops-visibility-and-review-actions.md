# 版本记录：v0.3-ops-visibility-and-review-actions

## 1. 本轮目标

在上一轮“三步工作台”已经跑通的基础上，继续补两类直接影响日常使用效率的能力：

- 让首页不仅能开始和停止，还能看见流程进度与近期关注事项
- 让审核修订区从“只看列表”升级为“可直接处理”

这轮目标属于：

- 功能补充
- 可维护性提升
- 使用效率优化

优先处理这件事的原因是，工作台入口虽然已经明确了，但如果用户点完开始后看不到系统在忙什么，或者进了审核区还要反复跳页，实际使用效率还是会被拖慢。

## 2. 背景与上下文

本轮承接：

- `custom/notes/v0.2-three-step-workbench.md`

上一轮已经把系统收束成明确的三步工作流：

- 开始工作
- 审核修订
- 停止

但继续从真实使用路径反推，会发现两个缺口：

1. 首页缺少“运行中的可见性”
2. 审核列表缺少“处理动作”

也就是说，系统已经有了主入口，但还不够像一个值班编辑真正依赖的控制台。

## 3. 问题分析

表面现象：

- 首页虽然能显示待处理数量，但缺少更直观的流程总览和最近发生了什么
- 审核修订区还是偏静态列表，只能看，不能在列表里快速处理
- 草稿审核动作依赖详情页，人工流程切换次数偏多

核心矛盾：

- 主流程入口已经收束，但执行反馈和处理效率还没有跟上

主要原因：

1. 前两轮优先保证工作流和后端闭环，首页更多承担“入口”职责
2. 审核逻辑已经存在，但缺少面向审核场景的快捷交互层
3. 草稿更新接口虽然支持审核动作，但默认 reviewer 绑定逻辑还不够适合快捷审核场景

## 4. 候选方案比较

### 方案 A：只补首页信息，不做审核快捷动作

核心思路：

- 首页增加更多统计卡片
- 审核区维持现状

优点：

- 改动小
- 风险低

缺点：

- 只能提升“看见问题”的能力，不能提升“处理问题”的效率

放弃原因：

- 用户从首页进入审核区后仍然需要多跳一步，收益不完整

### 方案 B：首页可见性 + 审核快捷动作一起补

核心思路：

- 首页增加流程进度总览和近期关注
- 审核修订区增加“去处理 / 通过 / 退回”快捷动作
- 草稿 PATCH 接口适配快捷审核场景

优点：

- 既提升了工作台可见性，也提升了人工处理效率
- 不需要推翻现有模型和页面结构

缺点：

- 需要同时修改首页、审核页和草稿接口

风险：

- 如果快捷审核没有正确带上 reviewer，会造成审核记录缺失

最终选择原因：

- 这是对上一轮工作台能力最直接、最有感知的增强

## 5. 最终决策

本轮最终采用方案 B，具体决策如下：

1. 首页增加“流程进度总览”
2. 首页增加“近期关注”，展示待修订稿件和最近动态
3. 审核修订区每行增加快捷处理动作
4. 草稿 PATCH 接口在快捷审核场景下默认把当前登录用户作为 reviewer
5. 审核动作继续复用既有草稿接口，不新增重复审核接口

## 6. 具体实现

本轮修改文件如下：

- `src/app/(dashboard)/dashboard/page.tsx`
- `src/app/api/drafts/[id]/route.ts`
- `src/app/(dashboard)/review/review-actions.tsx`
- `src/app/(dashboard)/review/page.tsx`
- `AGENTS.md`

各文件作用：

### `src/app/(dashboard)/dashboard/page.tsx`

- 在原有四张总览卡片下新增两块内容：
  - 流程进度总览
  - 近期关注
- 流程进度总览展示待抓取、待抽取、待草稿、待编辑、待审核、已通过等关键状态
- 近期关注展示：
  - 被退回修订的稿件
  - 最近操作日志

这样做的原因是，让值班人员不必跳出首页就能判断堵点和最近变化。

### `src/app/api/drafts/[id]/route.ts`

- 重写为更清晰的中文头部说明
- 在带 `reviewDecision` 的更新场景下，若请求未显式传 `reviewerId`，默认使用当前登录用户
- 审核动作写日志时补充 `status` 与 `reviewDecision` 细节

这一步是为了让审核列表里的快捷动作能安全复用现有接口，而不必再造一套新 API。

### `src/app/(dashboard)/review/review-actions.tsx`

- 新增审核区快捷操作组件
- 提供：
  - 去处理
  - 通过
  - 退回

“去处理”跳草稿详情页，“通过 / 退回”直接走接口处理。

### `src/app/(dashboard)/review/page.tsx`

- 在表格中新增“快捷处理”列
- 每一行可直接进入处理或快速完成审核动作

## 7. 本轮优点

本轮带来的主要正向效果：

- 首页从“能启动”变成“能观察”
- 审核修订区从“能查看”变成“能直接处理”
- 审核动作和详情页编辑之间的切换成本明显下降
- 草稿审核记录更适配快捷操作场景

得到改善的质量属性：

- 可用性
- 可维护性
- 协作效率
- 状态可见性

## 8. 本轮缺点与代价

本轮仍有明确限制：

1. 首页“近期关注”目前仍以日志和退回稿件为主，还不是完整的异常监控体系
2. 快捷通过 / 退回使用的是固定评论文案，后续仍应支持填写更具体的审核意见
3. 审核修订区目前还是列表页，不是卡片式待办工作台
4. 没有继续处理 Redis 版本偏低问题
5. 没有接入真实 OpenAI key 联调

这些代价当前是可接受的，因为本轮目标是提升日常操作效率，而不是扩大功能边界。

## 10. 本轮结论

本轮最重要的结论是：

三步工作台已经从“能控制系统开始与停止”进一步进入“能看见系统进展、能更快处理人工审核”的阶段。

被证实的假设：

- 不需要新增复杂模块，只靠在现有主流程上补可见性和快捷动作，就能明显提升使用体验
- 草稿详情接口经过小幅调整后，足以承载快捷审核场景

后续应默认继承的经验：

- 首页增强应优先补“值班判断能力”
- 审核相关增强应优先减少跳转次数

## 11. 下一轮建议

下一步最值得做的 1 到 3 件事：

1. 给审核快捷动作补“填写审核意见”的轻弹层
   - 收益最高
   - 能把快捷审核从“能处理”推进到“处理更完整”

2. 给工作台增加“任务失败 / 重试提示”
   - 风险最低
   - 能把近期关注从运营信息继续推进到异常处理

3. 配置真实 `OPENAI_API_KEY` 并做抽取、草稿生成联调
   - 对生产能力最关键
   - 需要外部配置后再推进
