# 版本记录：v0.5-review-comment-modal

## 1. 本轮目标

把审核意见输入从表格行内展开升级成更清晰的轻弹层，让审核动作更聚焦、更像正式操作，而不是在密集列表中临时展开一块输入区域。

这轮目标属于：

- 功能补充
- 可维护性提升
- 交互体验优化

优先做这件事的原因是，上一轮虽然已经支持填写审核意见，但行内展开会直接撑高表格行，视觉上不稳定，也不利于审核员集中处理当前稿件。

## 2. 背景与上下文

本轮承接：

- `custom/notes/v0.4-review-comments-and-queue-retry.md`

上一轮已经完成：

- 工作台异常与重试
- 审核意见输入

但审核意见输入仍是“在列表行里展开文本框”的方案。这个方案虽然能用，但继续观察后发现，它更像临时过渡态，不适合长期作为正式交互形态。

## 3. 问题分析

表面现象：

- 展开意见输入后，表格行会突然变高
- 列表上下文和审核动作混在一起，视觉焦点不够集中
- 多行文本和通过/退回按钮放在同一行区域里，操作感不够明确

核心矛盾：

- 审核动作已经具备了“完整信息输入”的需求，但当前承载方式还停留在轻度拼接

主要原因：

1. 上一轮优先补的是能力，不是最终交互形态
2. 当前项目没有现成 `Dialog` 组件
3. 如果直接引入完整弹窗体系，会扩大本轮改动范围

## 4. 候选方案比较

### 方案 A：继续保留行内展开，只微调样式

核心思路：

- 保留现有结构
- 通过间距和按钮位置优化减轻拥挤感

优点：

- 改动最小
- 不需要新增交互层

缺点：

- 根本问题没有变，表格仍然会因为展开而破坏阅读节奏
- 审核焦点仍然不够集中

放弃原因：

- 这只是延缓问题，不是真正解决问题

### 方案 B：在当前组件内实现轻量固定浮层

核心思路：

- 不新增完整通用 Dialog 体系
- 只在审核快捷组件中实现一个轻量固定浮层
- 浮层内完成意见填写与通过 / 退回

优点：

- 能快速获得更清晰的操作体验
- 保持改动边界小
- 不影响现有 API 和数据结构

缺点：

- 还不是完整的通用弹窗组件
- 复用性不如正式 UI 组件库中的 Dialog

最终选择原因：

- 这最符合“最小改动、解决当前真实问题”的原则

## 5. 最终决策

本轮最终采用方案 B，具体决策如下：

1. 保留“去处理”按钮在列表中直接可见
2. 把“审核意见”按钮改为打开固定浮层
3. 浮层内统一完成：
   - 审核说明
   - 意见填写
   - 取消
   - 退回修订
   - 审核通过
4. 支持 `Esc` 关闭
5. 打开浮层时锁定页面滚动，避免背景跟着滚动

## 6. 具体实现

本轮修改文件如下：

- `src/app/(dashboard)/review/review-actions.tsx`
- `AGENTS.md`

新增正式记录：

- `custom/notes/v0.5-review-comment-modal.md`

### `src/app/(dashboard)/review/review-actions.tsx`

主要改动：

- 移除行内 `Textarea`
- 新增轻量固定浮层
- 使用 `useEffect` 处理：
  - `Esc` 关闭
  - 打开弹层时锁定 `body` 滚动

交互变化：

- 列表中现在显示：
  - 去处理
  - 审核意见
- 点击“审核意见”后，在浮层里填写意见并执行通过或退回

这样做的好处是，列表继续保持紧凑，审核动作本身则获得了更清晰的操作上下文。

## 7. 本轮优点

本轮最大的提升是：

- 审核动作的视觉焦点更清晰
- 列表行高更稳定，不再因为意见输入突然拉高
- 审核通过 / 退回变得更像正式确认动作
- 改动范围仍然很小，没有牵动 API、schema 或其他业务页面

得到改善的质量属性：

- 可用性
- 可读性
- 交互稳定性
- 可维护性

## 8. 本轮缺点与代价

仍然存在这些限制：

1. 这还是页面内的轻量自实现浮层，不是通用 Dialog 组件
2. 没有加入更丰富的无障碍属性和焦点圈管理
3. 还没有支持把历史审核意见一并带进浮层上下文
4. 没有继续推进失败任务明细面板
5. 没有继续推进真实 OpenAI 联调

这些代价当前可接受，因为本轮目标本身就非常聚焦，只解决审核意见输入的交互清晰度问题。

## 10. 本轮结论

本轮最重要的结论是：

审核意见输入已经从“功能上可用”升级到“交互上更像正式审核动作”，而且是在不扩大系统复杂度的前提下完成的。

被证实的判断：

- 对当前项目来说，先做轻量定制浮层，比现在就引入完整弹窗体系更合适
- 审核区继续沿着“小范围高收益”的方式迭代是有效的

后续应默认继承的经验：

- 审核相关交互，优先保证焦点集中和列表稳定
- 在没有现成基础组件时，先用局部方案解决高频痛点，再决定是否抽象
