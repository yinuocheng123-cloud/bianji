# 版本记录：v0.10-content-detail-workbench

## 1. 本轮目标

本轮目标是补齐内容池的单条内容工作入口，让编辑不再只能在列表里批量看状态，而是能进入单条内容详情，直接完成抓取、抽取、草稿生成和结果核查。

这轮目标属于：
- 功能补充
- 可维护性提升

优先处理这个问题，是因为当前系统虽然已经有内容池、失败面板和批量操作，但缺少“单条内容工作页”会让人工排查和精细处理始终要绕路。

## 2. 背景与上下文

本轮承接：
- `custom/notes/v0.9-local-run-readiness.md`

修改前系统状态：
- 内容池只有列表页和批量操作
- 后端已经有单条内容读取接口
- 抓取、抽取、生成草稿的单条接口都已经存在
- 草稿已有详情页，但内容项没有对应详情页

因此这轮不需要改数据库和核心 API，而是适合围绕现有接口补一个真正可用的单条处理页。

## 3. 问题分析

表面现象：
- 内容池能看很多条内容，但点不进单条工作详情
- 出现抓取失败、抽取失败时，回到内容池后仍然要在列表里再次定位
- 编辑没法在一个页面同时查看原始 HTML、正文抽取结果、结构化结果和关联草稿

核心矛盾：
- 系统已有批量视角，但缺少单条处理视角

主要原因：
1. MVP 早期优先完成了整体流程骨架
2. 内容项的工作重心被放在批量推进上
3. 单条内容处理页尚未补齐，导致人工接管不顺

## 4. 候选方案比较

### 方案 A：继续只用内容池列表 + 批量按钮

核心思路：
- 不新增详情页，只在列表里继续扩操作

优点：
- 改动最少

缺点：
- 列表会越来越重
- 详情信息展示空间不足
- 原始 HTML、抽取正文、结构化结果很难在列表里舒适呈现

放弃原因：
- 这会让内容池列表承担过多职责，不利于后续继续迭代

### 方案 B：新增单条内容详情页，复用现有接口

核心思路：
- 新增 `/content-pool/[id]`
- 列表页增加“去处理”入口
- 在详情页中集中展示内容元信息、原始内容、抽取结果、结构化结果、来源记录和关联草稿
- 直接复用现有抓取、抽取、生成草稿接口

优点：
- 结构清晰
- 改动集中
- 不需要改 schema 和核心后端逻辑

缺点：
- 新增两个页面文件
- 首页失败面板还没有一起全部改成直达详情

最终选择原因：
- 这是当前最符合“最小必要改动 + 明显提升可用性”的方案

## 5. 最终决策

本轮采用方案 B。

关键决策如下：

1. 新增内容详情页，而不是继续把复杂信息塞进内容池列表
2. 详情页只复用现有内容读取、抓取、抽取、生成草稿接口，不新加后端协议
3. 列表页补单条入口，形成“批量推进 + 单条处理”双视角
4. 先把内容池这一条链路补通，不连带改首页失败面板，避免修改范围继续扩大

## 6. 具体实现

本轮修改文件如下：
- `src/app/(dashboard)/content-pool/page-client.tsx`
- `src/app/(dashboard)/content-pool/[id]/page.tsx`
- `src/app/(dashboard)/content-pool/[id]/page-client.tsx`
- `AGENTS.md`

### `src/app/(dashboard)/content-pool/page-client.tsx`

- 给内容标题增加详情链接
- 新增“去处理”操作列
- 保留现有批量抓取、抽取、生成草稿能力

### `src/app/(dashboard)/content-pool/[id]/page.tsx`

- 新增内容详情页
- 展示内容基础信息、关键词、原链接、抽取标题、抽取摘要、抽取正文
- 展示结构化提取结果和原始 HTML
- 展示关联草稿和来源记录

### `src/app/(dashboard)/content-pool/[id]/page-client.tsx`

- 新增单条操作栏
- 支持立即抓取、立即抽取、立即生成草稿
- 若草稿生成成功，直接跳转到对应草稿页

## 7. 本轮优点

当前方案相比之前更好的地方：

- 内容池从“只能看列表”升级成“可以单条处理”
- 人工排查时终于有稳定落点
- 单条页面把原始内容、抽取结果、结构化结果和草稿关系放在一起，减少来回切页
- 继续复用原有 API，避免后端再长出一套平行实现

得到改善的质量属性：
- 可用性
- 可维护性
- 调试便利性

## 8. 本轮缺点与代价

当前方案的已知限制：

1. 首页失败面板还没有全部改为直达内容详情
2. 内容详情页当前偏查看与推进，不是完整编辑表单
3. 详情页的操作提示还是轻量文本，不是完整通知体系
4. 这轮没有继续推进真实 AI 联调

这些代价当前可接受，因为本轮目标是先补上单条内容工作入口，而不是再扩一整套编辑能力。

## 9. 验证与结果

本轮完成的验证：

- `npm run lint`
- `npm run build`
- 登录后访问 `/content-pool`
- 从内容池页面提取一条内容详情链接并请求 `/content-pool/[id]`

验证结果：

- lint 通过
- build 通过
- 构建产物中出现 `/content-pool/[id]`
- 实测内容详情页返回 `200`

## 10. 本轮结论

本轮最重要的结论是：
内容池现在终于具备了单条内容工作台，内容处理链路比之前更接近真实编辑部使用方式。

被证实的判断：
- 现有单条接口已经足够支撑一个内容详情页
- 通过新增详情页而不是继续堆列表，是更稳妥的演进路径

后续应默认继承的经验：
- 列表负责筛选和批量推进
- 详情页负责单条排查和精细处理

## 11. 下一轮建议

下一步最值得做的 1~3 件事：

1. 把首页失败明细改成优先直达 `/content-pool/[id]`
   - 收益最高，能把异常处理进一步打通

2. 给内容详情页补“状态流转”和“负责人调整”
   - 风险低，能继续增强单条工作页价值

3. 配置真实 `OPENAI_API_KEY` 并联调 AI 抽取与草稿生成
   - 对生产能力提升最大，但需要外部配置条件
