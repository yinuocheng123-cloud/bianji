# 版本记录：v0.17-rule-logs-task-pause-manual-queue

## 1. 本轮目标

本轮目标是继续补强第二阶段 P0 运营底盘，把上一轮留下的三个高价值缺口补上：

- 规则中心的规则日志
- 任务中心的暂停 / 恢复
- 异常转人工后的承接队列

这一轮属于：
- 功能补充
- 可维护性提升
- 运营闭环增强

优先做这三件事，是因为上一轮已经把“任务详情、异常动作、规则编辑”做到了能用，但还没做到足够稳定和可追溯：
- 规则能改，但改完很难回看演化过程
- 任务能重试和取消，但还不能先停下来再恢复
- 异常能转人工，但转完之后还缺一个自然的接管落点

## 2. 背景与上下文

本轮承接：
- `custom/notes/v0.16-task-detail-exception-actions-rule-editor.md`
- `custom/notes/v0.15-stage2-task-exception-foundation.md`

修改前状态：
- `Task / TaskLog / ExceptionEvent / RuleDefinition` 已正式承接第二阶段底层
- `/ops/tasks/[id]` 已能查看任务详情并执行重试、取消
- `/ops/exceptions` 已能重跑、忽略、转人工、进入规则修正
- `/ops/rules` 已能新建、编辑、启停规则

但修改前仍存在的实际问题：
- 规则修改没有独立日志入口，难以回看版本演化
- 暂停中的任务无法在系统里明确表达，调度中枢能力不完整
- 异常“转人工”后只更新状态，没有可见的人工作业承接区

本轮继续遵守的原则：
- 不额外扩新的二阶段表，优先复用现有 `Task / ExceptionEvent / OperationLog`
- 继续走“最小增量、先闭环、后扩展”的节奏
- 不把这一轮扩成动作链、学习反馈或模板版本管理

## 3. 问题分析

表面看是缺按钮、缺日志页、缺状态入口，但真正的核心问题是：

- 第二阶段已经强调“可追踪、可中断、可回溯、可人工接管”
- 现有系统虽然已经有底层模型，却仍有几处“动作做了，但痕迹和承接不完整”的断点

当前最主要的三个断点是：

1. 规则修改不可回溯  
如果没有规则日志，规则中心就会慢慢退化成“大家都能改，但没人知道谁改了什么”的风险点。

2. 任务缺少缓冲控制  
只有重试和取消是不够的。对于小团队值班场景，暂停后再恢复是很常见、也很安全的中间动作。

3. 转人工后没有承接区  
如果异常转人工后只改一个状态，但没有稳定的集中列表，值班时仍然要靠人脑记忆，违背第二阶段的“少思考负担”目标。

## 4. 候选方案比较

### 方案 A：新增更多正式表

核心思路：
- 新增 `rule_logs`
- 新增 `manual_tasks`
- 进一步扩状态表和承接表

优点：
- 数据结构更标准
- 后续可扩展性更强

缺点：
- 会扩大 schema 改动面
- 需要同步改 seed、页面、API 和迁移验证
- 这轮的收益和风险比不够理想

放弃原因：
- 当前最重要的是把已有底层用起来，而不是继续铺新表

### 方案 B：复用现有 OperationLog 和 ExceptionEvent

核心思路：
- 用 `OperationLog` 承接规则日志
- 用 `Task.status = PAUSED` 承接暂停 / 恢复
- 用 `ExceptionEvent.status = MANUAL_PROCESSING` 承接人工接管队列

优点：
- 改动小
- 落地快
- 与现有架构一致
- 不需要数据库结构升级

缺点：
- 后续如果规则日志或人工任务变复杂，仍可能要拆出独立表
- 这轮的“人工承接”还不算完整人工作业系统

最终选择原因：
- 这是当前最符合“最小改动 + 立即可用”的方案

## 5. 最终决策

本轮最终采用方案 B。

关键设计决策如下：

1. 规则日志先复用 `OperationLog`  
先保证“谁在什么时候改了哪条规则”能被追溯，再决定后面是否拆成专门表。

2. 暂停 / 恢复只支持未开始执行的任务  
当前只对 `PENDING / RETRYING` 的任务开放暂停，对 `PAUSED` 开放恢复。运行中的任务暂不支持中断暂停，避免做出不真实的“假暂停”。

3. 人工接管先落在异常中心内  
先让 `MANUAL_PROCESSING` 状态有稳定展示区和负责人显示，不急着单独拆“人工任务中心”。

### 任务暂停 / 恢复

在 `src/lib/task-center.ts` 里新增和整理了：
- `enqueueTask`
- `pauseQueuedTaskById`
- `resumePausedTaskById`

并新增 API：
- `POST /api/tasks/[id]/pause`
- `POST /api/tasks/[id]/resume`

同时更新任务详情动作组件：
- `src/app/(dashboard)/ops/tasks/[id]/task-actions.tsx`

这一部分实现了：
- 未开始任务可以暂停
- 暂停任务可以恢复并重新入队
- 暂停和恢复都有日志留痕

这里特别做了一个取舍：
- 运行中的任务不支持中途暂停

这样虽然保守，但更真实，也更稳定。

### 规则日志

新增：
- `GET /api/rules/[id]/logs`

并在规则创建、规则更新时，把更完整的 `before / after` 信息写入 `OperationLog`。

同时修改：
- `src/app/(dashboard)/ops/rules/page.tsx`
- `src/app/(dashboard)/ops/rules/page-client.tsx`

现在规则中心里已经可以直接看到最近规则日志，知道：
- 谁改了规则
- 什么时候改的
- 是创建还是更新
- 版本是否发生变化

### 异常人工接管队列

修改：
- `src/app/api/exceptions/[id]/assign-manual/route.ts`
- `src/app/(dashboard)/ops/exceptions/page.tsx`

这部分实现了：
- 转人工后不再把异常“解决掉”，而是改成 `MANUAL_PROCESSING`
- 在异常中心新增“人工接管队列”
- 显示负责人、更新时间和建议下一步

这样一来，异常转人工后不再是一个消失的动作，而是进入可见的待处理区。

## 7. 本轮优点

相比上一轮，本轮让第二阶段 P0 的“运营闭环”更扎实了。

主要优点：

- 规则中心开始具备最小回溯能力
- 任务中心从“可重试 / 可取消”扩展到“可暂停 / 可恢复”
- 异常中心从“能转人工”扩展到“转人工后看得见、接得住”
- 继续复用现有模型，没有扩大 schema 风险

改善的质量属性：
- 可回溯性
- 可中断性
- 可接管性
- 可持续运营性

## 8. 本轮缺点与代价

这一轮仍有明显边界，不能夸大：

1. 规则日志仍复用 `OperationLog`  
这足够当前阶段使用，但如果后续要做规则版本对比、回滚和细粒度 diff，可能仍需独立表。

2. 暂停仍然不是“运行中暂停”  
当前只支持等待态任务暂停，不支持中途挂起 Worker 中的执行。

3. 人工接管队列仍然只是异常中心内的一段承接区  
还没形成独立的人工作业台、负责人分配和完成闭环。

4. 建议动作仍是轻量规则映射  
目前异常建议动作主要基于类型映射，不是更深的上下文推荐。

## 10. 本轮结论

本轮最重要的结论是：

第二阶段 P0 现在已经从“有基础处理动作”进一步推进到“有缓冲控制、可追日志、能接人工”的状态。

被再次证实的判断：
- 当前阶段继续复用既有底层模型，是最稳的推进方式
- 小团队运营系统最重要的不是模型多，而是“停得住、看得见、接得住”

本轮形成的新结论：
- 规则日志不一定要等独立表才能先落地
- 暂停 / 恢复是比“更复杂的自动化”更值得先补的调度中枢能力
- 人工接管队列必须可见，否则“转人工”只是表面动作

## 11. 下一轮建议

下一步最值得做的 1~3 件事：

1. 给异常中心补“人工接管完成”动作  
这样人工接手后能形成完整闭环，而不是一直停留在处理中状态。

2. 给规则日志补更清晰的变更摘要  
比如明确列出哪些字段被改了，而不只是显示版本变化。

3. 把人工接管队列逐步接到工作台首页  
这样值班人员一登录就能直接看到“今天谁该接哪些异常”。
