# 版本记录：v0.14-stage2-ops-p0-shell

## 1. 本轮目标

本轮目标是把第二阶段从“只有设计方案”推进到“系统里已经能看见、能进入、能继续承接开发”的第一批代码骨架。

本轮目标属于：
- 功能补充
- 可维护性提升
- 第二阶段基线落地

优先处理这个问题，是因为上一轮虽然已经完成了第二阶段完整架构设计，但系统里还没有真正出现 `/ops` 这层入口，导致第二阶段仍停留在文档层，无法让后续开发和实际使用围绕新结构继续演进。

## 2. 背景与上下文

本轮承接：
- `custom/notes/v0.13-stage2-ops-system-design.md`
- `custom/notes/v0.12-auxiliary-modules-ops-ready.md`

修改前状态：
- MVP 仍以“主流程页面 + 辅助模块页面”为主
- 第二阶段架构、模块、动作链、里程碑已经写入设计文档
- 但导航、路由、页面结构里还没有真正落出 `/ops`
- `AGENTS.md` 里的“最新一期正式记录”还停留在 `v0.12`

已知约束：
- 当前项目已经可运行，不能因为第二阶段升级破坏现有流程
- 仓库要求最小改动、先记录再扩展
- 当前阶段优先保证稳定和可继续迭代，不适合一次性把 Prisma 二阶段大 schema 全量迁移进去

## 3. 问题分析

表面问题：
- 第二阶段设计已经有了，但系统中没有对应入口
- 用户无法在界面里直观看到“任务中心、异常中心、规则中心、模板中心”等新系统层
- 后续继续开发时容易重新滑回“在原页面上继续堆功能”的路径

核心矛盾：
- 需要尽快把第二阶段结构做成真实代码
- 但又不能一上来就做高风险的大规模 schema 改造

主要原因：
1. 现有 MVP 已有生产性骨架，直接大迁移风险高
2. 第二阶段最需要先建立的是“系统层入口和模块边界”
3. 任务表、异常表、规则表、模板版本表等虽然重要，但应该在入口稳定后再逐步接入

## 4. 候选方案比较

### 方案 A：直接开始二阶段 Prisma 大迁移

核心思路：
- 一次性新增 tasks、exceptions、rules、ai_templates 等表
- 然后再补页面

优点：
- 数据层一步到位
- 后续 API 更完整

缺点：
- 改动面大
- 容易影响现有运行基线
- 当前页面和导航仍没有新系统层，用户短期看不到成果

放弃原因：
- 不符合当前“先稳住入口和结构，再逐步替换底层”的节奏

### 方案 B：先落 `/ops` 系统层入口和 P0 页面骨架，再逐步迁移 schema

核心思路：
- 先补导航和 `/ops`
- 先用现有数据库与日志数据做任务中心、异常中心、规则中心、模板中心的基础版页面
- 再在下一轮逐步落正式任务表、规则表等

优点：
- 风险小
- 可以立即把第二阶段结构显性化
- 后续每一轮开发都有稳定落点

缺点：
- 当前任务中心和异常中心仍属于“现有数据映射版”
- 还不是完整数据模型

最终选择原因：
- 这是最符合当前项目阶段的增量方案

## 5. 最终决策

本轮最终采用方案 B。

关键决策：
1. 先不改 Prisma schema
2. 先把 `/ops` 作为第二阶段系统层入口落地
3. 先完成 P0 最关键的四个页面骨架：
   - 任务中心
   - 异常中心
   - 规则中心
   - 模板中心
4. 同时补：
   - `/ops` 总览页
   - 流水线、学习反馈、资料沉淀、动作链的阶段页
5. 用现有 `OperationLog`、`PromptTemplate`、`Keyword`、`Site`、`Draft`、`ContentItem` 等数据先支撑骨架，不等待二阶段表全部就绪

- `src/app/(dashboard)/ops/knowledge/page.tsx`
- `src/app/(dashboard)/ops/chains/page.tsx`

各文件作用：

### `AGENTS.md`
- 把最新一期记录路径从 `v0.12` 更新到 `v0.13`
- 本轮结束后还会继续更新到 `v0.14`

### `src/lib/constants.ts`
- 为 `/ops` 增加导航入口
- 增加第二阶段模块说明文本
- 新增 `opsModules`，作为运营中枢总览页的模块配置来源

### `/ops` 总览页
- 作为第二阶段系统层统一入口
- 展示：
  - 待审核稿件
  - 异常待关注
  - 在线模板
  - 可运行来源
- 展示自动化阶段说明
- 展示 8 个模块入口卡片

### `/ops/tasks`
- 任务中心基础版
- 先基于 BullMQ 队列健康状态和 `OperationLog` 映射任务视图
- 展示：
  - 等待中
  - 执行中
  - 失败
  - 延迟
  - 最近任务执行记录

### `/ops/exceptions`
- 异常中心基础版
- 先把以下问题集中呈现：
  - `queue:failed`
  - 内容关键字段缺失
  - 审核前缺字段
- 错误说明尽量转成人话

### `/ops/rules`
- 规则中心基础版
- 先明确六大规则类型
- 先把现有可配置资产映射成规则基线：
  - 站点
  - 关键词
  - 提示词模板

### `/ops/templates`
- 模板中心基础版
- 先接入现有 `PromptTemplate`
- 说明当前能力边界与后续版本管理目标

### 其他阶段页
- `/ops/pipeline`
- `/ops/feedback`
- `/ops/knowledge`
- `/ops/chains`

这些页面当前主要作用是：
- 把第二阶段结构做成真实路由
- 避免导航里出现死概念或断链
- 为后续模块继续开发预留稳定路径

## 7. 本轮优点

相比上一轮只有设计文档的状态，本轮方案更好的地方在于：

- 第二阶段结构终于进入代码层，而不是停留在说明文档
- 用户现在登录后已经可以真正进入 `/ops`
- 后续开发不会再漂浮在“到底把功能挂到哪里”的不稳定状态
- 保持了现有 MVP 的稳定性，没有贸然做高风险 schema 大迁移

改善的质量属性：
- 可维护性
- 结构清晰度
- 后续扩展性
- 交互可感知性

## 8. 本轮缺点与代价

当前方案的已知缺点：

1. 任务中心仍是“队列状态 + 日志映射版”，不是正式 tasks 表
2. 异常中心仍是“失败日志 + 缺字段推断版”，不是正式 exceptions 表
3. 规则中心仍是“规则类型 + 现有配置基线映射”，不是正式 rules 表
4. 模板中心仍建立在 `PromptTemplate` 上，尚未接版本管理
5. 流水线、学习反馈、资料沉淀、动作链仍只是阶段页，还没深入业务实现

这些代价是刻意接受的，因为本轮目标不是“一步到位完成第二阶段”，而是先把系统层入口稳稳落地。

## 10. 本轮结论

本轮最重要的结论是：
第二阶段终于从“设计文档”推进到了“系统可访问入口”，而且是在不破坏现有 MVP 运行基线的前提下完成的。

被证实的判断：
- 先落系统层入口，再做 schema 迁移，是当前更稳的路径
- 用现有数据做 P0 页面骨架，足以支撑第二阶段继续滚动开发

后续应继承的经验：
- 第二阶段开发要继续坚持“先把结构接出来，再替换底层数据模型”
- 不要重新滑回“在原页面上零散堆功能”的路径

## 11. 下一轮建议

下一步最值得做的 1~3 件事：

1. 开始落二阶段 Prisma 基础表
   - 收益最高
   - 优先接：`tasks`、`task_logs`、`exceptions`、`rules`

2. 把 `/ops/tasks` 和 `/ops/exceptions` 从“映射版”升级成“正式数据版”
   - 风险中等
   - 能迅速提升第二阶段底盘的真实性

3. 给工作台首页接入第二阶段入口和推荐动作
   - 风险较低
   - 能让编辑/审核每天从第一屏就开始受益
