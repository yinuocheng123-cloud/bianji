# 版本记录：v0.19-manual-note-dashboard-links-rule-value-diff

## 1. 本轮目标

本轮目标是在 v0.18 的基础上，把人工接管和规则日志再往“值班可执行”推进一层，重点补三件事：

- 人工接管完成时记录处理说明
- 工作台首页的人工接管提醒增加直达处理入口
- 规则日志从字段级摘要升级到值级摘要

本轮属于：
- 功能补充
- 可维护性提升
- 值班体验增强

优先处理这三件事，是因为上一轮虽然已经补上了“人工完成”“首页提醒”“字段级规则日志”，但仍然存在三个明显不足：
- 异常处理完了，却不知道是怎么处理的
- 首页能看到人工接管提醒，但还不能直接点进去处理
- 规则日志知道改了哪些字段，但还看不出具体从什么改成什么

## 2. 背景与上下文

本轮承接：
- `custom/notes/v0.18-manual-complete-dashboard-hand-off-rule-diff.md`
- `custom/notes/v0.17-rule-logs-task-pause-manual-queue.md`

修改前状态：
- 异常中心已支持人工完成
- 工作台首页已能显示人工接管中的异常
- 规则日志已能显示字段级变更摘要

但修改前仍有三个痛点：
- 人工完成缺少处理说明
- 首页的人工接管提醒仍需再跳一次去找真正处理入口
- 规则日志无法展示变更前后值

## 3. 问题分析

当前问题已经不是“有没有功能”，而是“功能是否足够减少思考负担”。

主要矛盾有三点：

1. 人工接管缺少结果说明  
如果异常被人工关闭，但没有留下处理说明，后续复盘时仍然会失去上下文。

2. 首页提醒不够短路径  
对于值班台来说，看见提醒后还要再找入口，体验仍然偏重。

3. 规则日志缺少值级信息  
如果只知道“改了规则范围”，却看不到“从 A 改到 B”，日志价值会打折扣。

## 4. 候选方案比较

### 方案 A：继续只补展示，不补提交内容

核心思路：
- 首页只保留提醒
- 规则日志只保留字段列表
- 人工完成不补说明

优点：
- 改动最小

缺点：
- 仍然不能解决“处理完之后为什么这么处理”的问题
- 规则日志信息量不够

放弃原因：
- 不符合第二阶段“可追踪、可学习、可回溯”的方向

### 方案 B：补说明、补直达、补值级摘要

核心思路：
- 人工完成时提交处理说明
- 首页直接给人工接管入口
- 规则日志显示前后值

优点：
- 更贴近真实值班与复盘场景
- 不需要新表
- 性价比高

缺点：
- 会让页面逻辑稍微复杂一些

最终选择原因：
- 这是当前最能提升可执行运营感的一组改动

## 5. 最终决策

本轮最终采用方案 B。

关键设计决策如下：

1. 人工处理说明先写入 `ExceptionEvent.detailJson`  
这样不改 schema，也能把处理结果沉淀下来。

2. 首页提醒按异常上下文直接给去向  
优先跳内容工作区，其次跳任务详情，最后回异常中心。

3. 规则日志值级摘要先聚焦关键字段  
在当前字段级摘要基础上，补“从什么改成什么”，不急着做复杂 diff 组件。

## 6. 具体实现

本轮修改文件如下：

- `src/app/api/exceptions/[id]/complete-manual/route.ts`
- `src/app/(dashboard)/ops/exceptions/exception-actions.tsx`
- `src/app/(dashboard)/dashboard/page.tsx`
- `src/app/(dashboard)/ops/rules/page-client.tsx`
- `.gitignore`
- `AGENTS.md`

### 人工完成补处理说明

修改：
- `src/app/api/exceptions/[id]/complete-manual/route.ts`
- `src/app/(dashboard)/ops/exceptions/exception-actions.tsx`

现在人工完成时会：
- 弹出处理说明输入
- 把说明写入 `detailJson.manualResolutionNote`
- 同步写入操作日志

这样异常被人工关闭后，至少能知道处理人如何描述这次收口。

### 工作台首页补直达入口

修改：
- `src/app/(dashboard)/dashboard/page.tsx`

现在人工接管提醒会根据异常上下文给出最短路径：
- 如果有 `contentItemId`，直达内容工作区
- 如果关联任务，直达任务详情
- 否则回异常中心

这样首页提醒不再只是“告诉你有异常”，而是“告诉你去哪处理”。

### 规则日志补值级摘要

修改：
- `src/app/(dashboard)/ops/rules/page-client.tsx`

现在规则日志不仅会告诉你改了哪些字段，还会展示：
- 规则名称：旧值 -> 新值
- 规则范围：旧值 -> 新值
- 启停状态：停用 -> 启用
- 备注：旧值 -> 新值
- 规则内容：旧 JSON -> 新 JSON

### Git 忽略补充

修改：
- `.gitignore`

把本地运行产生的：
- `*.out.log`
- `*.err.log`

加入忽略，避免首次提交时把开发日志一并推上去。

## 7. 本轮优点

本轮最大的提升是：人工处理和规则回溯开始真正具备“后续能复盘”的价值，而不只是“当前能点完”。

主要优点：
- 人工接管完成后有处理说明
- 工作台首页能直接带人去处理入口
- 规则日志终于开始具备值级可读性
- 没有扩大 schema 风险

改善的质量属性：
- 可回溯性
- 可复盘性
- 值班效率
- 运营连续性

## 8. 本轮缺点与代价

这一轮仍然有边界：

1. 人工处理说明仍然是轻量文本  
还没做结构化处理结果。

2. 首页直达入口是启发式映射  
目前是按异常上下文推断最佳入口，不是更复杂的推荐引擎。

3. 规则日志值级摘要在 `ruleContentJson` 上仍然偏粗  
会显示 JSON 字符串，而不是细粒度 JSON diff。

本轮完成的验证包括：

- `npm run build`

验证结果：
- lint 通过

本轮未重复执行的验证：

## 10. 本轮结论

本轮最重要的结论是：

第二阶段 P0 现在已经不只是“能处理人工接管”和“能看规则日志”，而是开始有了更像正式运营系统的复盘痕迹。

被再次证实的判断：
- 小团队后台最重要的是“看见提醒后马上能处理、处理完还能知道为什么这样处理”
- 规则日志只要继续往人话和值级方向推进，就能显著提升后续维护效率

## 11. 下一轮建议

下一步最值得做的 1~3 件事：

1. 给人工处理说明接入异常中心详情展示  
这样关闭后的异常也能直接回看处理结论。

2. 给工作台人工接管提醒补负责人过滤  
让每个人先看到自己该接的异常。

3. 给规则日志中的 `ruleContentJson` 补更友好的 JSON 摘要  
避免大段原始 JSON 影响可读性。
