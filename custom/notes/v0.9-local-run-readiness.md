# 版本记录：v0.9-local-run-readiness

## 1. 本轮目标

本轮目标不是继续扩展功能，而是把当前系统拉到“本机执行 `npm run dev` 后可以直接测试使用”的状态，并完成最小必要的运行验证。

这轮目标属于：
- 可维护性提升
- 运行验证

之所以优先做这件事，是因为前几轮已经把系统功能骨架、工作台、队列、审核和 OAuth 代码路径搭好了，但如果本地运行状态没有被确认，用户仍然无法直接进入测试。

## 2. 背景与上下文

本轮承接：
- `custom/notes/v0.8-google-oauth-login.md`

修改前系统状态：
- Next.js 开发环境、数据库、Redis 配置已经存在
- 密码登录链路已经可用
- Google OAuth 代码链路已经接入，但尚未配置 Google 客户端参数
- Worker 已具备启动条件

本轮不再改动功能实现，重点转为确认当前环境是否真的可以打开、登录、进入工作台。

## 3. 问题分析

表面问题是：用户要求“下一步就 `npm run dev`，可以直接测试使用”。

核心矛盾不是功能缺失，而是运行态是否稳定：
- `3000` 端口是否正常监听
- PostgreSQL / Redis 是否已启动
- 演示账号是否还能登录
- 登录后是否能进入工作台
- 队列控制接口在登录态下是否可正常访问

如果这些基础链路有任何一项未打通，系统虽然“代码存在”，但仍不具备直接测试价值。

## 4. 候选方案比较

### 方案 A：继续补代码，假定运行没有问题

核心思路：
- 不做运行验证，直接告诉用户执行 `npm run dev`

优点：
- 速度最快

缺点：
- 风险最高
- 一旦环境变量、端口、数据库或登录链路有问题，用户会再次卡住

放弃原因：
- 这会违反“不要假装完成”的原则

### 方案 B：不改功能代码，直接完成运行态验证

核心思路：
- 先确认 PostgreSQL / Redis
- 启动 `npm run dev`
- 启动 `npm run workers`
- 校验 `/login`
- 校验密码登录
- 校验登录后 `/dashboard`
- 校验登录态下 `/api/control/status`

优点：
- 能最直接回答“现在是否可测”
- 不引入额外功能风险

缺点：
- 不会新增功能
- 只能确认当前环境，不代表其他机器无需配置即可运行

最终选择原因：
- 这最符合本轮任务目标

## 5. 最终决策

本轮采用方案 B，只做运行态拉起和最小必要验证，不继续扩功能。

关键决策有三点：

1. 使用现有 `.env.local`，不新增新的本地配置分支
2. 以密码登录作为当前可测主入口，因为 Google OAuth 仍缺少外部客户端配置
3. 同时启动开发服务器和 Worker，保证三步工作台相关能力具备基础运行条件

## 6. 具体实现

本轮没有修改业务代码，只进行了环境启动与验证，并同步补充本记录。

执行与确认内容如下：

- 执行 `git fetch --all --prune`
- 读取 `AGENTS.md`
- 读取 `custom/notes/v0.8-google-oauth-login.md`
- 检查 `.env.local`
- 确认 PostgreSQL 监听 `5432`
- 确认 Redis 监听 `6379`
- 后台启动 `npm run dev -- --hostname 127.0.0.1 --port 3000`
- 后台启动 `npm run workers`
- 校验 `http://127.0.0.1:3000/login`
- 使用演示账号调用 `/api/auth/login`
- 带登录态访问 `/dashboard`
- 带登录态访问 `/api/control/status`

另外，为了方便后续排查，启动日志输出到了：
- `custom/runtime-logs/dev.out.log`
- `custom/runtime-logs/dev.err.log`
- `custom/runtime-logs/worker.out.log`
- `custom/runtime-logs/worker.err.log`

## 7. 本轮优点

本轮最大的价值在于把“代码可构建”推进到了“当前机器上可直接测试”。

相比只看 lint / build，本轮额外确认了：
- 页面可打开
- 演示账号可登录
- 登录后能进入工作台
- 队列状态接口在登录态下可访问

得到改善的质量属性：
- 可用性
- 调试便利性
- 交接清晰度

## 8. 本轮缺点与代价

这轮也有明确限制：

1. 没有新增任何业务能力
2. Google OAuth 依然不能实际登录，因为本机尚未配置：
   - `GOOGLE_CLIENT_ID`
   - `GOOGLE_CLIENT_SECRET`
3. `OPENAI_API_KEY` 仍为空，真实 AI 抽取与草稿生成还不能联调
4. 当前验证仅针对本机环境，不代表其他机器无需配置即可运行

这些代价当前是可接受的，因为本轮目标本来就不是扩能力，而是确认当前环境的可测试性。

## 10. 本轮结论

本轮最重要的结论是：
当前项目已经处于“本机执行开发环境后可直接测试”的状态，不需要再先修运行链路。

被证实的判断：
- `.env.local`、数据库、Redis、开发服务器、Worker、密码登录和工作台状态接口已形成可用闭环

仍未改变的判断：
- Google OAuth 仍然只是代码就绪，不是环境就绪
- 真实 AI 能力仍需要单独配置 `OPENAI_API_KEY`
