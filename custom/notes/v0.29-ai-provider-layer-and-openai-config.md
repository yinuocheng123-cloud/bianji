# 版本记录：v0.29-ai-provider-layer-and-openai-config

## 1. 本轮目标

本轮目标是把项目从“各模块各自直接调用 OpenAI”推进到“统一 AI Provider 接入层 + AI 配置中心 + 调用日志”的形态。

这一轮优先级高，原因有三个：

1. 用户已经明确纠正方向：`OAuth` 在这里更应该理解为模型接入与授权能力，而不是人登录后台。
2. 当前企业资料检索、结构化抽取、草稿生成三条链路各自直连，后续很难统一追踪、统一切换模型、统一排错。
3. 第二阶段要往“可持续自动运行、可追踪、可学习”的系统靠拢，没有统一 AI 调用日志，就无法支撑后面的学习反馈和质量复盘。

本轮属于：

- 功能补充
- 结构收口
- 可维护性提升

## 2. 背景与上下文

本轮承接：

- `custom/notes/v0.28-company-review-categories-and-proof-view.md`
- `custom/notes/v0.27-company-feedback-and-evidence-preview.md`

修改前的状态是：

1. `src/lib/ai.ts` 直接实例化 OpenAI，用于结构化抽取和草稿生成。
2. `src/lib/company-discovery.ts` 也直接实例化 OpenAI，用于企业资料检索。
3. 系统里没有正式的 AI 提供商配置表。
4. 系统里没有正式的 AI 调用日志表。
5. 提示词模板有独立管理页，但没有版本号，后续无法稳定记录“本次调用用了哪一版模板”。

已知约束：

- 不把真实 `OPENAI_API_KEY` 存进数据库。
- 保持当前 Next.js / Prisma / PostgreSQL / Redis / BullMQ 技术栈不变。
- 企业资料检索允许保留 fallback；结构化抽取和草稿生成则应在缺失真实 OpenAI 条件时显式失败。

## 3. 问题分析

表面现象是：

1. AI 功能已经“能用”，但没有统一入口。
2. 配置散在 `.env` 和代码里，运营侧看不到当前到底在用什么模型。
3. 调用失败时只能从接口报错里看，无法回答“最近失败的是哪一条链路、用了哪个模板、耗时多久”。

真正的核心矛盾是：

- 第二阶段强调“自动化流程可追踪、可复盘、可持续学习”，而现有 AI 链路还停留在“能调通就算完成”的工具阶段。

当前瓶颈主要是：

1. 没有统一 Provider 层，导致三条业务链路重复维护。
2. 没有 AI 调用日志，导致反馈中心和异常中心无法进一步接 AI 质量数据。
3. 没有模板版本，导致无法可靠记录“输入模板版本”。
4. 真实 `OPENAI_API_KEY` 当前仍为空，所以真实 OpenAI 联调必须和代码接入分开看待。

## 4. 候选方案比较

### 方案 A：只在现有 `ai.ts` 和 `company-discovery.ts` 上补日志

核心思路：

- 保留现有分散调用方式，只在每个函数里各自写调用日志。

优点：

- 改动小。
- 能较快补出“有日志”。

缺点：

- 提供商配置还是分散的。
- 后续切换模型或新增别的 Provider 仍然要改多处。
- 无法形成真正的“AI 接入层”。

放弃原因：

- 这只是给旧结构打补丁，不符合这轮“统一 Provider 层”的目标。

### 方案 B：新增统一 AI Provider 层，三条业务链统一走一套调用入口

核心思路：

- 新增 `AIProviderConfig` 和 `AICallLog`。
- 新建统一 `src/lib/ai-provider.ts`。
- 企业资料检索、结构化抽取、草稿生成全部从该层发起调用。

优点：

- 模型、Base URL、启停状态、模板版本、耗时和错误全部能统一管理。
- 后续接更多模型能力或 AI 反馈统计时更稳定。
- 更符合第二阶段“操作系统化”的方向。

缺点：

- 需要同时动 schema、seed、服务层、设置页和几条业务链路。
- 这轮真实 OpenAI 联调仍然会被空密钥阻塞。

最终选择原因：

- 这是当前收益最高、后续复用价值最大的方案。

## 5. 最终决策

本轮最终采用方案 B，并做了以下关键决策：

1. 新增 `AIProviderConfig`，但只保存非敏感配置，不存真实 API Key。
2. 新增 `AICallLog`，统一记录：
   - provider
   - scenario
   - model
   - duration
   - status
   - error
   - inputTemplateVersion
   - inputVariables
   - targetType / targetId
3. 给 `PromptTemplate` 增加 `version` 字段，并在更新模板时自动递增。
4. 企业资料检索继续保留 fallback，但真实 AI 尝试失败时也会记录失败日志。
5. 结构化抽取和草稿生成切到统一 Provider 层后，不再各自直接 new OpenAI。
6. 新增后台页 `/settings/ai`，统一查看当前 AI Provider 配置与最近调用日志。

### 1. 数据层

在 `prisma/schema.prisma` 中新增：

- `AIProviderType`
- `AICallScenario`
- `AICallStatus`
- `AIProviderConfig`
- `AICallLog`

并给 `PromptTemplate` 增加：

- `version`

### 2. 统一 AI Provider 层

在 `src/lib/ai-provider.ts` 中新增了统一能力：

- 读取 OpenAI 当前启用配置
- 解析默认模型
- 渲染模板变量
- 调用 OpenAI Responses API
- 写入 AI 调用日志

这里刻意做了两点约束：

1. API Key 仍只来自环境变量
2. 配置中心虽然保留 `reasoningEffort` 字段，但本轮没有强行下发到请求参数里

第二点是一个主动取舍：当前 SDK 类型约束对 reasoning 参数更严格，而本轮更重要的是先把真实链路和日志层稳定落下来。

### 3. 三条业务链统一切换

`src/lib/ai.ts`

- 结构化抽取改走统一 Provider
- 草稿生成改走统一 Provider

`src/lib/company-discovery.ts`

- 企业资料检索优先走统一 Provider
- 调用失败时自动回退到公开检索兜底
- 即使回退，真实 AI 尝试的失败也会进入调用日志

`src/lib/pipeline.ts`

- 把操作人 `createdById` 继续向下传，便于 AI 调用日志记录触发人

### 4. 配置中心与后台入口

新增：

- `/settings/ai`
- `/api/ai/providers/openai`
- `/api/ai/calls`

配置中心现在可以看到：

- 提供商状态
- 是否检测到 `OPENAI_API_KEY`
- 默认模型配置
- Base URL
- 最近 AI 调用日志

### 5. 模板版本

在 `src/app/api/prompts/[id]/route.ts` 中：

- 每次 PATCH 自动把 `version + 1`

在 `src/app/(dashboard)/settings/prompts/page-client.tsx` 中：

- 补了模板版本显示

## 7. 本轮优点

相比之前，本轮最大的提升是：

1. AI 接入开始真正“系统化”了，不再是三个模块各自直连。
2. 运营和管理员第一次能在后台直接看见模型配置和最近调用情况。
3. 模板版本正式进入调用日志，后续复盘更可靠。
4. 企业资料检索、结构化抽取、草稿生成三条链已经可以共享同一套 Provider 与日志口径。
5. PostgreSQL 和 Redis 在本机重新拉起，开发验证环境回到可运行状态。

改善的质量属性包括：

- 可维护性
- 可追踪性
- 可运营性
- 可扩展性

## 8. 本轮缺点与代价

这一轮也有明确代价：

1. 真实 `OPENAI_API_KEY` 当前仍为空，所以不能声称已经完成真实 OpenAI 联调。
2. 企业资料检索虽然切到了统一 Provider 层，但当前真实运行仍然会 fallback。
3. `reasoningEffort` 目前先只做配置保留，没有实际下发到 OpenAI 调用参数中。
4. Prisma 在 Windows + PostgreSQL 环境下的个别历史表和大小写问题仍然需要注意，seed 本轮为此做了更保守的写法。
5. build 期间 Redis 未启动时会有连接噪音，因此本轮特地把 Redis 重新拉起做运行态验证。

## 10. 本轮结论

本轮最重要的结论是：

“大模型调用接入层 + AI 配置中心 + 统一调用日志”已经从设计意图进入了实际可运行的代码层。

被证实的判断：

1. AI 接入层不能继续散在业务模块里，否则第二阶段后续迭代会越来越难。
2. 模板版本如果不进日志，后面无法稳定复盘输出质量。
3. 对企业资料检索这类链路，保留 fallback 是必要的，但 fallback 也必须在统一日志体系下被看见。
4. 当前最需要继续推进的，不是登录 OAuth，而是真实模型调用接入与配置治理。

## 11. 下一轮建议

下一步最值得继续做的 1~3 件事：

1. 在你提供有效 `OPENAI_API_KEY` 后，直接完成真实 OpenAI 联调，并记录真实 AI 调用样本。
2. 把 AI 调用日志接进学习反馈中心，形成按场景的成功率 / 失败率统计。
3. 给 AI 配置中心补“手动测试一次模板”的最小沙盒能力。
