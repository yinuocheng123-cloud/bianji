# 版本记录：v0.7-failure-links-and-categories

## 1. 本轮目标

把工作台里的“最近失败明细”继续推进到可处理程度：

- 给失败记录增加失败类型分类
- 给失败记录增加最短处理入口

这轮目标属于：

- 功能补充
- 可维护性提升
- 异常处理闭环增强

优先做这件事的原因是，上一轮虽然已经能在首页看到最近失败发生在哪，但值班编辑仍然需要自己判断“这属于哪类失败”“下一步该点去哪儿”。这一步如果不补，异常面板仍然停留在信息展示层。

## 2. 背景与上下文

本轮承接：

- `custom/notes/v0.6-recent-failure-details.md`

上一轮已经完成：

- 最近失败明细
- 完整日志入口

继续从值班视角看，下一步最自然的增强不是再加更多统计，而是把“看见问题”推进到“点一下就能去处理问题”。

## 3. 问题分析

表面现象：

- 最近失败明细里虽然已经有队列名和报错信息
- 但值班编辑仍然要自己判断是抓取失败、抽取失败还是草稿失败
- 也仍然要自己决定应该去内容池、草稿详情还是日志页

核心矛盾：

- 失败信息已经足够，但缺少最后一层“操作指引”

主要原因：

1. 上一轮优先解决的是“看见失败”
2. 工作台还没有把失败类型和下一步动作做成可点击结构
3. 对于草稿相关失败，系统还没有自动帮用户找到最相关的草稿入口

## 4. 候选方案比较

### 方案 A：只给失败加分类，不加入口

核心思路：

- 根据队列名映射失败类型
- 用标签显示

优点：

- 改动小
- 风险低

缺点：

- 用户还是得自己找入口
- 只解决了“看懂”，没有解决“去处理”

放弃原因：

- 这轮目标已经明确是往可处理程度推进，只做分类不够

### 方案 B：分类 + 处理入口一起补

核心思路：

- 根据队列名映射出失败类型标签
- 为抓取/抽取失败提供内容池入口
- 为草稿失败优先查找对应草稿详情，没有草稿时回退到草稿中心

优点：

- 真正把异常面板从信息展示推进到处理入口
- 不需要新增 API
- 完全可以复用已有页面结构

缺点：

- 首页服务端逻辑比上一轮略复杂，需要补一层失败内容到草稿的映射

最终选择原因：

- 这是当前最贴近“可应用程度”的增强方式

## 5. 最终决策

本轮最终采用方案 B，具体决策如下：

1. 按队列名把失败分成：
   - 抓取失败
   - 抽取失败
   - 草稿失败
   - 流程失败
2. 给每条失败加 `Badge`
3. 抓取 / 抽取失败默认指向内容池
4. 草稿失败优先根据 `contentItemId` 查找最新草稿并直达详情页
5. 找不到草稿时回退到草稿中心

## 6. 具体实现

本轮修改文件如下：

- `src/app/(dashboard)/dashboard/page.tsx`
- `AGENTS.md`

新增正式记录：

- `custom/notes/v0.7-failure-links-and-categories.md`

### `src/app/(dashboard)/dashboard/page.tsx`

主要改动：

- 新增 `getFailureMeta()`，基于队列名决定：
  - 分类标签
  - 标签颜色
  - 默认跳转入口
  - 操作文案
- 对最近失败记录做二次处理
- 根据 `contentItemId` 查询最新草稿映射
- 草稿失败优先直达 `/drafts/[id]`
- 列表里新增：
  - 类型标签
  - 处理按钮

这样做的效果是，值班编辑在首页看到失败后，不需要先理解内部实现细节，也能快速进入最相关的处理页面。

## 7. 本轮优点

本轮带来的主要提升：

- 最近失败明细从“信息卡片”变成了“待处理入口”
- 失败类型更直观，减少判断成本
- 草稿类失败可以更快进入对应稿件处理
- 继续复用现有页面结构，没有引入额外复杂度

得到改善的质量属性：

- 可用性
- 可观察性
- 处理效率
- 可维护性

## 8. 本轮缺点与代价

本轮仍然存在这些限制：

1. 失败类型分类目前基于队列名规则，不是更深层语义分类
2. 抓取 / 抽取失败仍然是回到内容池，而不是直达某条内容的单页详情
3. 草稿映射现在默认取最新草稿，不处理更复杂的一对多场景
4. 还没有继续推进真实 AI 联调
5. Redis 版本建议升级问题仍在

这些代价当前可以接受，因为本轮重点是把异常卡片变成可操作入口，而不是重构整个内容详情体系。

## 10. 本轮结论

本轮最重要的结论是：

三步工作台里的异常处理已经从“看见失败”推进到“能马上去处理失败”，首页控制台的实用性又上了一个台阶。

被证实的判断：

- 在当前架构下，不需要新接口也能把异常处理进一步落地
- 基于队列名和内容 ID 做轻量规则映射，已经足够支撑当前阶段的失败处理入口

后续应默认继承的经验：

- 首页失败信息增强优先考虑“下一步点去哪”
- 在还没有内容详情单页前，优先复用内容池和草稿详情页面承载处理动作

## 11. 下一轮建议

1. 接入真实 `OPENAI_API_KEY` 并做 AI 抽取、草稿生成联调

3. 补内容单条详情页

   - 但改动范围会比前两项大
