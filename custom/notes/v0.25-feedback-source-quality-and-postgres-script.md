# 版本记录：v0.25-feedback-source-quality-and-postgres-script

## 1. 本轮目标

本轮目标继续沿着 v0.24 的收口方向推进，但不扩新 schema，而是补两类“立刻可用”的能力：

1. 在学习反馈中心补上来源质量观察，让运营人员能看到哪些来源更值得提权、哪些来源应该降权。
2. 把本机 PostgreSQL 的启动和停止收束成可提交、可重复执行的脚本，减少重启机器后的人工恢复成本。

本轮属于：

- 可维护性提升
- 运营能力补充
- 运行环境可重复化

## 2. 背景与上下文

本轮承接：

- `custom/notes/v0.24-postgres-feedback-grouped-rule-diff.md`

上一轮已经解决了：

- 本机 PostgreSQL 恢复
- 学习反馈中心基础统计
- 人工完成标签接入反馈口径
- `ruleContentJson` 分组 diff

但继续往运营层面看，还有两个短板：

1. 反馈中心虽然能看到审核和人工收口，但还看不到“来源质量”这一类长期最重要的运营判断信号。
2. PostgreSQL 虽然已经跑起来，但恢复动作还是手动命令，缺少可直接重复执行的脚本。

## 3. 问题分析

表面现象是：

1. 当前反馈中心仍主要围绕审核和异常，没有把内容来源的质量差异转成视图。
2. 本机 PostgreSQL 已经可用，但重启后如何恢复仍然依赖记忆和手工命令。

真正的核心矛盾是：

- 第二阶段已经进入“可持续运行”的要求，来源质量是后续规则优先级和自动化分配的重要基础，不能一直缺席。
- 本机环境如果不能脚本化恢复，就很难称为稳定的开发与演示底盘。

## 4. 候选方案比较

### 方案 A：继续只补页面，不处理本机启动脚本

核心思路：

- 只在反馈中心里增加来源排行
- PostgreSQL 启停继续靠手工命令

优点：

- 页面推进更快
- 改动更少

缺点：

- 本机恢复流程仍然脆弱
- 新人或后续代理继续接手时，还要重新猜命令

放弃原因：

- 和“可持续运行”的目标不一致。

### 方案 B：同时补来源质量排行和 PostgreSQL 启停脚本

核心思路：

- 反馈中心从 `ContentItem.source + status` 聚合来源质量信号
- PostgreSQL 启动、初始化、补库和停止动作封装进脚本

优点：

- 同时增强运营层可读性和本机环境可重复性
- 改动小，但价值高

缺点：

- 会新增 `custom/` 下的脚本和说明文档

最终选择原因：

- 这是当前成本最低、最符合第二阶段“连续运转”方向的方案。

## 5. 最终决策

本轮最终采用方案 B，关键决策如下：

1. 反馈中心来源质量不引入新表，先基于现有 `ContentItem` 的 `source + status` 做轻量质量评分。
2. PostgreSQL 启停脚本放在 `custom/postgres/`，而不是放到被忽略的 `custom/runtime/`，这样既能提交到仓库，又不会污染业务目录。
3. 启动脚本做成幂等：已初始化则直接启动，缺库则自动补 `editorial`。

## 6. 具体实现

本轮改动文件如下：

- `src/app/(dashboard)/ops/feedback/page.tsx`
- `custom/postgres/start-local-postgres.ps1`
- `custom/postgres/stop-local-postgres.ps1`
- `custom/postgres/README.md`
- `custom/notes/v0.25-feedback-source-quality-and-postgres-script.md`
- `AGENTS.md`

### 1. 反馈中心补来源质量排行

在 `src/app/(dashboard)/ops/feedback/page.tsx` 中：

- 新增 `summarizeSourceQuality`
- 以来源维度聚合：
  - 总量
  - 通过数
  - 驳回数
  - 归档数
  - 待审核数
- 通过轻量评分生成：
  - 高质量来源排行
  - 低质量来源观察

这一步先把来源质量信号做成可运营视图，后续再继续升级成正式评分模型。

### 2. PostgreSQL 启动脚本

新增 `custom/postgres/start-local-postgres.ps1`：

- 自动定位当前仓库下的 PostgreSQL 运行文件
- 首次缺失时自动 `initdb`
- 启动本机 `5432`
- 自动确认 `editorial` 数据库存在

这样后续本机恢复时不需要再手输一串命令。

### 3. PostgreSQL 停止脚本

新增 `custom/postgres/stop-local-postgres.ps1`：

- 直接调用 `pg_ctl stop`
- 避免手动找进程或误关其他服务

### 4. 使用说明

新增 `custom/postgres/README.md`：

- 说明启动命令
- 说明停止命令
- 说明运行目录、数据目录和日志目录

## 7. 本轮优点

本轮的主要提升有：

1. 学习反馈中心开始覆盖来源质量这个长期价值很高的运营信号。
2. PostgreSQL 恢复流程被固定成脚本，降低了机器重启后的恢复成本。
3. 新增文件都集中在 `custom/` 下，符合仓库约束，不干扰业务结构。

改善的质量属性包括：

- 可维护性
- 运营可读性
- 环境可重复性

## 8. 本轮缺点与代价

本轮也有明显边界：

1. 来源质量评分目前仍是轻量启发式，不是正式来源评分模型。
2. 脚本是本机开发/演示脚本，不是系统服务安装器。
3. 反馈中心仍未接入模板效果、字段修改频率和来源评分回写逻辑。

本轮执行了以下验证：

- `git fetch --all --prune`

- `npm run lint`
- `npm run build`
- 登录态访问 `/ops/feedback`

验证结果：

1. PostgreSQL 启动脚本可重复执行。
2. `lint` 通过。
3. `build` 通过。
4. `/ops/feedback` 正常返回。

## 10. 本轮结论

本轮最重要的结论是：

第二阶段的“可持续运行”不只是自动链和异常中心，还包括来源质量这种运营信号，以及本机环境的可重复恢复能力。

被再次证明有效的方向包括：

- 先用现有数据做轻量评分视图，比等完整评分系统更适合持续迭代。
- 启停脚本这种小工具，往往比继续堆页面更能提升真实可用性。

1. 把反馈中心继续补模板效果观察口径。
