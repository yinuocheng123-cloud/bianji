# 版本记录：v0.16-task-detail-exception-actions-rule-editor

## 1. 本轮目标

本轮目标是把第二阶段 P0 底盘继续从“已有正式表和基础列表”推进到“开始具备操作闭环”的状态。

这一轮优先处理的内容属于：
- 功能补充
- 可维护性提升
- 结构升级

优先做这件事，是因为上一轮虽然已经把 `tasks / task_logs / exceptions / rules` 正式落库，但页面层仍然存在三个明显断点：
- 任务中心只能看列表，无法进入单条任务追踪
- 异常中心能看到异常，但还不能做忽略、转人工、进入规则修正等动作
- 规则中心有正式规则表，但还没有最小创建/编辑/启停闭环

如果这几个断点不补，第二阶段虽然有了底层数据，却还称不上真正可操作的运营底盘。

## 2. 背景与上下文

本轮承接：
- `custom/notes/v0.15-stage2-task-exception-foundation.md`
- `custom/notes/v0.14-stage2-ops-p0-shell.md`

修改前状态：
- `/ops/tasks` 已切到正式 `Task / TaskLog` 数据
- `/ops/exceptions` 已切到正式 `ExceptionEvent` 数据
- `/ops/rules` 已切到正式 `RuleDefinition` 数据
- 抓取、抽取、草稿生成链路已开始写正式任务与异常记录

但修改前仍存在的实际问题：
- 值班人员无法在任务中心里继续追踪一条任务的完整执行过程
- 异常中心仍偏“展示页”，无法直接形成处置闭环
- 规则中心仍偏“只读台”，无法承接第二阶段“规则可配置、不写死”的目标

本轮继续遵守的约束：
- 不推翻上一轮已落地的二阶段基础表
- 不在这一轮铺开动作链、反馈中心、模板版本管理
- 继续采用“先把高频操作闭环打通，再扩更深层结构”的增量方式

## 3. 问题分析

表面问题是页面里缺按钮、缺详情、缺表单，但真正的核心矛盾是：

- 第二阶段系统层已经开始强调“可追踪、可中断、可重试、可修正规则”
- 但实际界面和 API 还停留在“能看见数据”的阶段，没有形成“看见后马上处理”的动作闭环

当前瓶颈主要有三类：

1. 任务追踪断点  
任务列表里能看到任务存在，但无法进一步查看它的负载、重试情况、步骤日志和关联对象，导致任务中心还不够像真正的调度中枢。

2. 异常处理断点  
异常中心能识别失败，但处理动作仍然要靠用户自己再跳去其他页面或重新找入口，违背了“2-3 人小团队高频使用”的设计原则。

3. 规则修正断点  
规则中心虽然已有正式规则表，但没有基本编辑能力，意味着系统还不能把“发现问题 -> 修正规则”真正落到后台操作层。

这些问题里，第一和第二类更接近“操作系统闭环缺口”，优先级最高；第三类虽然偏配置，但它是第二阶段“逐步从死功能升级为可调教系统”的关键入口，也必须补上最小闭环。

## 4. 候选方案比较

### 方案 A：继续只补 API，不补页面闭环

核心思路：
- 先继续扩任务、异常、规则 API
- 页面等后面统一再做

优点：
- 改动更集中在服务端
- 风险较小

缺点：
- 用户仍然无法直接在后台完成处理动作
- 第二阶段的“运营感”提升不明显
- 容易继续积累“底层有了、界面还没跟上”的脱节

放弃原因：
- 不符合当前“高频操作闭环优先”的目标

### 方案 B：同步补任务详情、异常动作和规则最小编辑闭环

核心思路：
- 给任务中心补详情页和取消动作
- 给异常中心补忽略、转人工、进入规则修正
- 给规则中心补创建、编辑、启停能力

优点：
- 能直接形成值班和运营闭环
- 更符合第二阶段“可追踪、可中断、可修正”的系统目标
- 改动仍然集中在 P0 模块，范围可控

缺点：
- 需要同时改页面、API 和部分任务状态逻辑
- 会进一步提高 `tasks / exceptions / rules` 三者的耦合，需要后续继续整理 orchestration 分层

最终选择原因：
- 这是当前最贴近“可执行运营层面”的增量升级方式

## 5. 最终决策

本轮最终采用方案 B。

关键设计决策如下：

1. 任务中心先补“详情 + 取消 + 重试”三件最关键能力  
不急着上更复杂的暂停/恢复，先让任务具备最基础的追踪与中断能力。

2. 异常中心优先补人工处置动作  
用“重跑、忽略、转人工、规则修正”四个动作形成最小闭环，而不是继续只做异常展示。

3. 规则中心先做轻量 JSON 编辑器  
不直接上复杂可视化规则编排器，先用文本 JSON + 基本字段表单承接规则配置，保证尽快可用、便于后续迭代。

4. 继续沿用上一轮的数据模型  
本轮不改 schema，不做新表，重点把已有 `Task / ExceptionEvent / RuleDefinition` 用起来。

## 7. 本轮优点

相比上一轮，本轮最明显的提升是第二阶段 P0 模块开始真正“能处理事情”，而不只是“能看到数据”。

具体优点包括：

- 任务中心从列表页升级到可追踪单条任务
- 异常中心从展示页升级到处置入口
- 规则中心从只读页升级到可配置入口
- 值班路径更短，减少了人工跳转和重复寻找入口
- 更符合“小团队高频协作”的设计方向

改善的质量属性：
- 可追踪性
- 可中断性
- 可操作性
- 可维护性
- 运营连续性

## 8. 本轮缺点与代价

这一轮的已知缺点和代价如下：

1. 任务中心还没有补暂停/恢复  
当前先落了“取消 + 重试”，但更完整的调度控制还没做完。

2. 异常动作仍是轻量闭环  
“转人工”和“规则修正”目前主要是状态推进与跳转入口，还不是完整的人工作业编排系统。

3. 规则编辑器目前是 JSON 文本方式  
对非技术用户来说已经比直接改代码好很多，但还不够完全可视化。

4. 规则修改日志还没落  
虽然规则已可编辑，但这轮还没把规则历史和差异记录补上，后续仍需继续完善。

5. 任务详情页主要承接已有任务数据  
后续如果动作链和流水线更复杂，任务详情页还需要继续接入更多上下文，比如链执行记录、异常关系和人工接管痕迹。

这些妥协当前是接受的，因为本轮目标是先补高频闭环，而不是一次性把第二阶段全部做满。

## 10. 本轮结论

本轮最重要的结论是：

第二阶段 P0 现在已经从“有正式底层”进一步推进到了“有正式操作闭环”的状态。

被进一步证实的判断：
- 第二阶段正确的推进顺序，仍然是“先数据承接，再补高频操作闭环”
- 小团队运营后台最重要的不是菜单多，而是看见问题后能立刻处理

本轮形成的新结论：
- 任务详情页、异常动作、规则编辑器是第二阶段继续往动作链和工作台升级前的必要桥梁
- 规则中心的最小可用方案可以先从 JSON 配置开始，不必一开始就追求复杂可视化

## 11. 下一轮建议

下一步最值得做的 1~3 件事：

1. 给规则中心补规则日志  
这是当前收益最高的下一步，因为规则已经能改，接下来必须能追溯“谁改了什么”。

2. 给任务中心补暂停 / 恢复  
这是当前风险最低、又能继续强化调度中枢能力的一步。

3. 给异常中心补“转人工后的人工作业承接”  
包括人工待办、负责人和建议动作，能更自然接到工作台。
