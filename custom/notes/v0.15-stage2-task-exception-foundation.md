# 版本记录：v0.15-stage2-task-exception-foundation

## 1. 本轮目标

本轮目标是把第二阶段 P0 从“只有 `/ops` 页面骨架”推进到“已经有真实底层数据承接”的状态。

本轮目标属于：
- 功能补充
- 结构升级
- 可维护性提升

优先处理这个问题，是因为上一轮虽然已经把第二阶段系统层入口做出来了，但任务中心、异常中心、规则中心仍主要依赖现有运行时数据映射，缺少正式模型支撑，无法持续积累和回放。

## 2. 背景与上下文

本轮承接：
- `custom/notes/v0.14-stage2-ops-p0-shell.md`
- `custom/notes/v0.13-stage2-ops-system-design.md`

修改前状态：
- `/ops` 已可访问
- `/ops/tasks` 主要看队列状态和旧日志
- `/ops/exceptions` 主要看 `queue:failed` 和字段缺失推断
- `/ops/rules` 还没有正式规则表
- 抓取、抽取、草稿生成仍没有同步写入正式任务表

已知约束：
- 不能破坏现有抓取、抽取、草稿生成链路
- 不适合这轮就把动作链、模板版本、反馈中心全部铺开
- 当前最值得先落的是任务、异常、规则三条底层主线

## 3. 问题分析

表面问题：
- 第二阶段页面已经出现了，但底下没有真正的数据实体
- 自动任务执行完之后，系统无法形成正式任务记录
- 失败虽然能进日志，但异常中心还没有正式异常表承接
- 规则中心还没有规则表，仍停留在“现有配置映射”层

核心矛盾：
- 页面层已经开始按第二阶段设计组织
- 数据层还没跟上，导致“操作系统化”只完成了一半

主要原因：
1. 第一阶段核心链路是围绕内容处理做的，没有独立任务模型
2. 第二阶段任务、异常、规则要和现有链路兼容，不能直接推翻
3. 需要一套增量方案，把现有主链逐步接进新模型

## 4. 候选方案比较

### 方案 A：只新增表，不改现有业务入口

核心思路：
- 把 `tasks`、`exceptions`、`rules` 加进 schema
- 但现有 `crawl / extract / generate-draft` 先不接

优点：
- 代码改动更少

缺点：
- 表会空着
- 页面仍然只能继续靠旧数据映射
- 第二阶段底层没有真正开始运转

放弃原因：
- 不符合“让系统连续跑起来”的目标

### 方案 B：新增表 + 让现有主链路开始写新表

核心思路：
- 先加最小基础表：`tasks`、`task_logs`、`exceptions`、`rules`
- 再把现有抓取、抽取、草稿生成入口和 worker 事件接进去
- 最后让 `/ops` 页面优先读取正式表

优点：
- 第二阶段底层开始真实运转
- 现有业务入口无需推倒重来
- 后续继续扩异常中心、工作台、动作链都有稳固落点

缺点：
- 改动跨 schema、API、worker、页面
- 需要同步处理数据库初始化和种子数据

最终选择原因：
- 这是当前最合理的增量升级方式

## 5. 最终决策

本轮最终采用方案 B。

关键决策：
1. 先只加四类基础实体：
   - `tasks`
   - `task_logs`
   - `exceptions`
   - `rules`
2. 不在这一轮就铺开动作链、反馈表、模板版本表
3. 现有 `crawl / extract / generate-draft` 直接开始写任务表
4. Worker 完成/失败事件直接更新任务状态并写入异常表
5. `/ops/tasks`、`/ops/exceptions`、`/ops/rules` 优先切到新表读取

### `prisma/schema.prisma`

新增了：
- 枚举：
  - `TaskType`
  - `TaskStatus`
  - `TaskTriggerType`
  - `ExceptionType`
  - `ExceptionSeverity`
  - `ExceptionStatus`
  - `RuleType`
- 模型：
  - `Task`
  - `TaskLog`
  - `ExceptionEvent`
  - `RuleDefinition`

同时给 `User` 增加了新关系，便于后续查看任务创建人、规则创建人、异常处理人。

### `src/lib/task-center.ts`

新增统一服务层，负责：
- 创建任务
- 写任务日志
- 更新任务状态
- 创建异常记录
- 任务重入队
- 队列失败与异常类型映射

这样避免在多个 API 和 worker 里重复写落库逻辑。

### 现有内容处理 API

修改：
- `src/app/api/content-items/[id]/crawl/route.ts`
- `src/app/api/content-items/[id]/extract/route.ts`
- `src/app/api/content-items/[id]/generate-draft/route.ts`

新增能力：
- 即时执行前创建任务
- 入队前创建任务
- 入队后记录 queue job id
- 即时失败写入异常表
- 返回结果中带 `taskId`

### `src/workers/start.ts`

补了任务和异常联动：
- Worker 接手任务时更新为 `RUNNING`
- 成功时更新为 `SUCCESS`
- 失败时更新为 `FAILED`
- 失败时同步创建 `ExceptionEvent`

这一步很关键，因为它让“任务中心”和“异常中心”不再只是人工触发时写数据，而是和自动流程真正连起来了。

### 新增 API

新增：
- `GET /api/tasks`
- `POST /api/tasks`
- `GET /api/tasks/[id]/logs`
- `POST /api/tasks/[id]/retry`
- `GET /api/exceptions`
- `POST /api/exceptions/[id]/retry`
- `GET /api/rules`
- `POST /api/rules`
- `GET /api/rules/[id]`
- `PATCH /api/rules/[id]`

当前用途：
- 为第二阶段页面和后续快捷动作提供正式数据接口
- 先支持查询、创建、重试、更新这些最基础动作

### `/ops` 页面切换

修改：
- `src/app/(dashboard)/ops/tasks/page.tsx`
- `src/app/(dashboard)/ops/exceptions/page.tsx`
- `src/app/(dashboard)/ops/rules/page.tsx`

变化：
- 任务中心优先读 `Task` / `TaskLog`
- 异常中心优先读 `ExceptionEvent`
- 规则中心开始读 `RuleDefinition`

### `prisma/seed.mjs`

补了可重复执行的二阶段种子数据：
- 示例规则
- 示例失败任务
- 示例任务日志
- 示例异常记录

同时处理成可重复 seed，不会每次都无限叠加脏数据。

## 7. 本轮优点

这一轮相比上一轮更好的地方：

- 第二阶段底层开始真实落库，不再只是页面外壳
- 现有抓取/抽取/草稿生成主链路已经接入正式任务记录
- 自动失败开始有正式异常表承接
- 规则中心不再只有概念，已经有规则表作为正式入口
- 页面现在开始能自然切向新模型，而不是继续依赖临时映射

改善的质量属性：
- 可追踪性
- 可重试性
- 结构稳定性
- 后续扩展性

## 8. 本轮缺点与代价

已知缺点：

1. 这轮只落了任务、异常、规则的基础版，还没接动作链、反馈中心、模板版本
2. `/ops/exceptions` 现在虽然已有正式异常表，但异常处理动作还比较轻
3. `/ops/rules` 已有正式规则表，但还没做规则日志表和可视化编辑器
4. `/ops/tasks` 还没有任务详情页，只能看列表和日志接口
5. Worker 与任务中心的耦合开始增加，后续需要继续抽一层更清晰的 orchestration 层

这些代价当前是可接受的，因为这轮重点是把最关键的 P0 底层先接起来。

## 10. 本轮结论

本轮最重要的结论是：
第二阶段已经真正拥有第一批“会积累、会留痕、会失败落库、会重试”的正式底层，而不是只有页面结构。

被证实的判断：
- 先接最小基础表，再把现有主链路逐步接上，是可行且稳定的
- 现有 MVP 不需要推翻，也能平滑长成第二阶段操作系统

后续应继承的经验：
- 继续沿用“先接底层数据，再切页面”的顺序
- 继续避免一次性铺开太多二阶段模型

## 11. 下一轮建议

下一步最值得做的 1~3 件事：

1. 给任务中心补详情页和任务操作
   - 包括暂停、取消、查看完整日志
   - 收益最高

2. 给异常中心补正式动作
   - 忽略
   - 转人工处理
   - 一键进入规则修正

3. 给规则中心补最小编辑闭环
   - 创建表单
   - 更新表单
   - 启停控制
   - 规则修改日志
