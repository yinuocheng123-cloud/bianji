# 版本记录：v0.1-mvp-baseline

## 1. 本轮目标

从空仓库建立“整木网AI编辑部”的首个可运行 MVP，并继续补到增强版：

- PostgreSQL / Redis 真正跑起来
- 内容池、草稿、审核、企业资料库形成闭环
- AI 改成真实模型接入路径
- 补 Worker、批量操作和更细权限

## 2. 背景与上下文

本仓库起始为空仓库，因此首轮需要同时完成：

- 技术基线
- 数据模型
- 页面与接口
- 首份正式记录

在此基础上，本轮又继续推进到更接近商用的增强版。

## 3. 问题分析

仅有页面和接口并不等于可用后台。真正要能落地，需要同时满足：

- 数据库真实可用
- 队列真实可用
- AI 接口具备真实替换路径
- 抓取链路可扩展到动态站点
- 权限边界不能继续停留在“登录即全可改”

## 4. 候选方案比较

### 方案 A：保留占位 AI 和同步接口

优点：

- 改动小

缺点：

- 不适合继续往商用方向推进
- 无法验证真实 AI 与异步链路

### 方案 B：继续补真实 AI、Worker 与批量操作

优点：

- 更接近真实编辑部工作流
- 后续接 Playwright、真实模型和更细权限时不需要推翻骨架

缺点：

- 环境依赖增加
- Redis / Playwright / OpenAI 配置需要更完整

最终选择方案 B。

## 5. 最终决策

本轮实际采取的增强决策：

- 启动本机 PostgreSQL 与 Redis
- 执行 `db:push` 和 `db:seed`
- `src/lib/ai.ts` 切换为 OpenAI Responses API 路径
- `src/lib/scrape.ts` 支持 Playwright 优先抓取
- 新增 BullMQ Worker 入口
- 新增内容池与草稿中心批量操作
- 新增接口级角色权限校验

## 6. 具体实现

### 基础设施

- PostgreSQL 已在本机 `5432` 启动
- Redis 已在本机 `6379` 启动
- 已完成数据库同步与种子数据导入

### 真实 AI

- `src/lib/ai.ts`
  - 改为真实 OpenAI 调用
  - 使用 JSON Schema 保证结构化输出
  - 提示词仍从数据库模板读取

### 抓取与流水线

- `src/lib/scrape.ts`
  - 支持 `auto / browser / http` 三种模式
- `src/lib/pipeline.ts`
  - 统一抓取、抽取、草稿生成流水线
- `src/workers/start.ts`
  - 消费 BullMQ 队列

### 批量操作

- `POST /api/content-items/batch`
- `POST /api/drafts/batch`
- 内容池和草稿中心均新增批量操作面板

### 权限

- 管理员、编辑、审核员、访客已做导航与接口级区分
- 提示词模板仅管理员可改
- 审核相关更新收紧到审核角色

## 7. 本轮优点

- 后台已经不再只是骨架，而是真正连上数据库与队列
- AI 已切换到真实服务接入路径
- 抓取链路开始具备动态站点支持能力
- 批量操作和权限边界更符合多人协作后台场景

## 8. 本轮缺点与代价

- 仍然需要真实 `OPENAI_API_KEY` 才能完成线上 AI 联调
- Playwright 浏览器若未安装，会回退到普通 HTTP 抓取
- 当前 Redis 版本偏旧，BullMQ 给出升级建议
- 审核通知、失败重试、抓取告警等仍未补齐

## 10. 本轮结论

本轮最重要的结论是：

这套系统已经从“可展示的 MVP”进入“可继续商用增强的后台基线”。
