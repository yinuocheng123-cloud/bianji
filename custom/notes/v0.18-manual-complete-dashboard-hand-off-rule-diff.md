# 版本记录：v0.18-manual-complete-dashboard-hand-off-rule-diff

## 1. 本轮目标

本轮目标是在 v0.17 的基础上，继续补强第二阶段 P0 的“人工接管”和“规则回溯”能力，让系统更贴近小团队真实值班场景。

这一轮优先处理的事项是：
- 异常中心补“人工接管完成”动作
- 工作台首页接入人工接管提醒
- 规则日志改成更清楚的字段变更摘要

本轮属于：
- 功能补充
- 可维护性提升
- 运营体验增强

之所以优先做这三件事，是因为上一轮虽然已经把“转人工”“规则日志”“暂停恢复”补出来了，但还存在三个明显不完整点：
- 转人工后只能看见在处理中，却没有明确的“处理完成”收口动作
- 人工接管提醒还停留在异常中心内部，没有进入工作台首页
- 规则日志虽然能看版本变化，但还不够直观，不能快速看出改了哪些核心字段

## 2. 背景与上下文

本轮承接：
- `custom/notes/v0.17-rule-logs-task-pause-manual-queue.md`
- `custom/notes/v0.16-task-detail-exception-actions-rule-editor.md`

修改前状态：
- `/ops/exceptions` 已能转人工，并有人工接管队列
- `/dashboard` 仍未直接显示人工接管中的异常
- `/ops/rules` 已能展示规则日志，但摘要仍偏粗

当前阶段的真实使用目标仍然没变：
- 服务 2-3 人高频值班团队
- 优先降低思考负担和跳转成本
- 强调人工审核与人工接管，而不是盲目追求全自动

## 3. 问题分析

本轮核心问题不是“缺少更多模块”，而是现有闭环的最后几步还不够顺。

主要问题有三类：

1. 人工接管没有完成动作  
如果人工已经处理完一条异常，但系统里没有正式“完成”入口，这条异常就只能长期挂在处理中状态，导致状态污染。

2. 工作台缺少人工接管可见性  
异常中心里虽然有人工接管队列，但用户每天首先打开的是工作台首页。如果首页看不到这些提醒，仍然不够符合值班习惯。

3. 规则日志缺少直接可读性  
只显示“版本从 v1 到 v2”还不够。对于非技术值班人员来说，更重要的是一眼看出“改了规则名称、规则范围还是规则内容”。

## 4. 候选方案比较

### 方案 A：继续只补后端，不改页面

核心思路：
- 只增加人工完成 API
- 页面先不动

优点：
- 改动更小
- 逻辑更集中

缺点：
- 使用体验提升有限
- 仍然不能在首页形成值班提醒
- 规则日志的人话程度不够

放弃原因：
- 不符合“页面适合 2-3 人小团队高频使用”的阶段目标

### 方案 B：同时补动作、入口和摘要

核心思路：
- 给异常中心补人工完成
- 给工作台补人工接管提醒
- 给规则日志补字段级摘要

优点：
- 形成更自然的人工处理闭环
- 值班入口更集中
- 日志更容易看懂

缺点：
- 改动同时涉及异常中心、工作台和规则中心
- 仍然没有独立人工任务系统

最终选择原因：
- 这是当前最平衡、最贴近真实运营场景的方案

## 5. 最终决策

本轮最终采用方案 B。

关键设计决策如下：

1. 人工完成先落在异常中心  
继续沿用 `ExceptionEvent.status`，由 `MANUAL_PROCESSING` -> `RESOLVED` 形成闭环。

2. 工作台首页直接接人工接管区  
先不新建独立页面，优先让值班首页成为真正的“今日该处理什么”入口。

3. 规则日志摘要直接做字段名人话映射  
不做复杂 diff 组件，先给出“哪些关键字段改了”的清晰摘要和标签。

## 6. 具体实现

本轮修改文件如下：

- `src/app/api/exceptions/[id]/complete-manual/route.ts`
- `src/app/(dashboard)/ops/exceptions/exception-actions.tsx`
- `src/app/(dashboard)/ops/exceptions/page.tsx`
- `src/app/(dashboard)/dashboard/page.tsx`
- `src/app/(dashboard)/ops/rules/page-client.tsx`
- `AGENTS.md`

### 异常人工完成

新增：
- `POST /api/exceptions/[id]/complete-manual`

并重写：
- `src/app/(dashboard)/ops/exceptions/exception-actions.tsx`

现在异常动作支持：
- 重跑
- 转人工
- 人工完成
- 规则修正
- 忽略

这样人工接管后的异常终于有了正式收口动作。

### 异常中心承接增强

修改：
- `src/app/(dashboard)/ops/exceptions/page.tsx`

现在：
- 开放异常区会按状态传入动作组件
- 人工接管区也可以直接点“人工完成”

也就是说，异常中心现在已经不只是“把转人工的异常堆在一起”，而是能在同一页里完成接手和收口。

### 工作台首页接入人工接管提醒

修改：
- `src/app/(dashboard)/dashboard/page.tsx`

新增了“人工接管提醒”模块，显示：
- 当前人工处理中的异常
- 负责人
- 最近更新时间

这样用户一登录进入工作台，就能直接知道哪些异常今天该接。

### 规则日志字段级摘要

重写：
- `src/app/(dashboard)/ops/rules/page-client.tsx`

现在规则日志支持：
- 自动识别 `before / after`
- 比较 `ruleName / ruleType / ruleScope / ruleContentJson / isActive / remark`
- 用中文字段名展示“改了哪些项”
- 同时用 Badge 展示关键变更字段

这让规则日志从“能看版本变化”升级成“能看清这次具体改了什么”。

## 7. 本轮优点

这轮最大的提升是把“人工接管”和“规则回溯”都往更实用、更像值班系统的方向又推了一步。

主要优点：
- 异常人工处理终于有了明确完成动作
- 工作台首页开始真正承担“人工接管提醒”职责
- 规则日志更容易被非技术用户理解
- 没有引入新表，仍然保持改动克制

改善的质量属性：
- 状态完整性
- 工作台可见性
- 日志可读性
- 值班效率

## 8. 本轮缺点与代价

这一轮仍然有清晰边界：

1. 人工完成仍只是异常层闭环  
还没有更完整的人工任务分配、完成说明和二次审核流。

2. 工作台里的人工接管提醒还比较轻  
当前主要是提醒和列表，不是完整的人工作业面板。

3. 规则日志摘要是字段级，不是值级 diff  
现在能看出“改了什么字段”，但还没直接展示“从什么改成了什么”。

## 10. 本轮结论

本轮最重要的结论是：

第二阶段 P0 的“人工接管”闭环现在更完整了，工作台也开始真正承担“值班首页”的职责。

被再次证实的判断：
- 对小团队来说，最值钱的不是更复杂的自动化，而是“系统能告诉你今天该接什么、处理完后怎么收口”
- 规则日志只要先做到足够清楚，就已经能显著减少后续维护成本

本轮形成的新结论：
- 人工接管提醒应该尽早进入工作台，而不是只留在异常中心
- 规则日志摘要必须说人话，否则就算有日志也很难在运营层真正用起来

## 11. 下一轮建议

下一步最值得做的 1~3 件事：

1. 给工作台的人工接管提醒增加直达处理入口  
让首页可以直接跳到对应异常或关联内容。

2. 给规则日志补值级摘要  
在重要字段上显示“从什么改成什么”。

3. 给人工接管完成补处理说明  
避免异常虽然被关闭，但后续不知道是怎么处理掉的。
