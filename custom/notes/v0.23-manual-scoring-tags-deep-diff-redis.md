# 版本记录：v0.23-manual-scoring-tags-deep-diff-redis

## 1. 本轮目标

本轮目标围绕第二阶段 P0 值班闭环继续收口，重点解决四件事：

1. 把人工接管优先级从轻量启发式标签升级成更稳定的评分结构。
2. 给人工完成动作补上“处理结果标签”，让异常收口不再只有一段自由文本。
3. 继续细化 `ruleContentJson` 的深层键变化摘要，减少规则变更复盘成本。
4. 恢复本机 Redis，让 BullMQ 队列与自动链路重新进入可运行状态。

本轮属于：
- 功能补充
- 可维护性提升
- 运行环境修复

## 2. 背景与上下文

本轮承接：
- `custom/notes/v0.22-start-work-button-motion.md`
- `custom/notes/v0.21-manual-links-priority-rule-key-diff.md`

上一轮已经把：
- 人工接管入口做成了首页可见
- 人工完成记录补了直达入口
- `ruleContentJson` 做到了顶层键级摘要

但距离“更接近商用值班台”还有三个明显短板：

1. 首页优先级依然偏启发式，缺少更稳定的数值结构。
2. 人工完成只写说明，没有结果标签，后续统计与学习反馈不好接。
3. 规则日志只能看顶层键，不够覆盖嵌套对象和数组。

同时，本机 Redis 之前处于不可用或低版本状态，已经直接影响 BullMQ worker 和自动链稳定性。

## 3. 问题分析

表面现象分别是：

1. 首页人工接管提醒虽然有高、中、低，但排序依据还不够稳定。
2. 异常“人工完成”之后，只能看到一句说明，无法直接区分是补数据、修规则还是核查后忽略。
3. `ruleContentJson` 的日志虽然比早期可读，但仍然难以回答“深层哪个键变了”。
4. 本机虽然一度能把 6379 端口拉起来，但旧版 Windows Redis 只有 `3.0.504`，BullMQ 明确要求 `>= 5.0.0`，所以“端口开了”并不等于“队列真恢复了”。

真正的核心矛盾是：
- 值班台不只是看见信息，而是要减少判断负担。
- 自动链不只是代码存在，而是运行环境真的要满足要求。

## 4. 候选方案比较

### 方案 A：继续沿用现有标签式优先级和文本说明

核心思路：
- 首页继续按 `high / medium / low` 排序
- 人工完成继续只记录说明
- 规则日志继续做顶层键摘要

优点：
- 改动小
- 风险低

缺点：
- 仍然过度依赖人工二次判断
- 学习反馈中心后续难以沉淀结构化经验
- 深层规则变化仍然不够直观

放弃原因：
- 这条路只能“勉强能用”，达不到本轮想要的值班稳定性。

### 方案 B：引入评分结构、结果标签和递归路径摘要

核心思路：
- 用数值分 + 标签理由共同表达优先级
- 人工完成同时记录 `resultTag + note`
- 规则日志把嵌套对象拍平成路径再做差异摘要
- Redis 直接切到兼容 BullMQ 的 7.x 兼容实现

优点：
- 更适合后续学习反馈和推荐动作
- 页面排序更稳定
- 规则变更更接近“可复盘”
- 自动链运行环境真正恢复

缺点：
- 改动点跨越页面、接口和本机运行环境
- 本机 Redis 的长期服务化仍可能受系统权限影响

最终选择原因：
- 这是当前最能同时提升“运营可执行性”和“运行稳定性”的方案。

## 5. 最终决策

本轮最终采用方案 B。

关键决策如下：

1. 首页人工接管优先级改成“评分结构 + 标签 + 原因”三层表达。
2. 人工完成时强制收集结果标签，但说明仍保持可选，以兼顾高频场景和结构化沉淀。
3. `ruleContentJson` 的差异不再只看顶层，而是递归展开为路径级变化。
4. 本机 Redis 不再停留在旧版 `3.0.504`，改用 Memurai Developer for Valkey 运行文件，实测版本为 `7.2.4`。

## 6. 具体实现

本轮修改文件如下：

- `src/app/(dashboard)/dashboard/page.tsx`
- `src/app/(dashboard)/dashboard/page-client.tsx`
- `src/app/(dashboard)/ops/exceptions/exception-actions.tsx`
- `src/app/(dashboard)/ops/exceptions/page.tsx`
- `src/app/(dashboard)/ops/rules/page-client.tsx`
- `src/app/api/exceptions/[id]/complete-manual/route.ts`
- `.gitignore`
- `custom/notes/v0.23-manual-scoring-tags-deep-diff-redis.md`
- `AGENTS.md`

### 1. 首页人工接管评分结构

在 `src/app/(dashboard)/dashboard/page.tsx` 中：
- 把原先轻量优先级判断改成数值评分
- 评分来源包括：
  - 是否已明确分配负责人
  - 是否涉及核心自动链路
  - 是否直接关联任务执行
  - 积压时长

输出结果改为：
- `priority`
- `priorityLabel`
- `priorityScore`
- `priorityReason`

在 `src/app/(dashboard)/dashboard/page-client.tsx` 中：
- 首页人工接管提醒改为按 `是否分配给我 -> 分数 -> 更新时间` 排序
- 页面直接展示评分数字和原因

### 2. 人工完成结果标签

在 `src/app/api/exceptions/[id]/complete-manual/route.ts` 中：
- 新增 `resultTag` 入参
- 允许的标签包括：
  - `FIXED_DATA`
  - `FIXED_RULE`
  - `HANDLED_MANUALLY`
  - `IGNORED_AFTER_CHECK`
  - `OTHER`
- 将标签写入 `detailJson.manualResolutionTag`
- 日志中同步记录 `manualResolutionTag`

在 `src/app/(dashboard)/ops/exceptions/exception-actions.tsx` 中：
- 去掉原来简单的 prompt
- 改成轻量浮层
- 浮层里同时完成：
  - 选择结果标签
  - 填写处理说明
  - 提交人工完成

在 `src/app/(dashboard)/ops/exceptions/page.tsx` 中：
- 人工处理中和最近人工完成区都补上了结果标签回显

### 3. 深层规则键变化摘要

在 `src/app/(dashboard)/ops/rules/page-client.tsx` 中：
- 新增递归对象展开逻辑
- 将嵌套对象与数组拍平成路径形式
- 变更摘要现在支持：
  - 新增路径
  - 删除路径
  - 变更路径
- 同时保留 before -> after 的简短值级预览

### 4. Redis 运行环境恢复

运行环境层面做了两段尝试：

1. 先验证了旧版 Windows Redis 压缩包方案  
   结果：
   - 6379 可以拉起
   - 但版本是 `3.0.504`
   - BullMQ worker 明确报错，不能继续使用

2. 改为 Memurai Developer for Valkey 4.3.2  
   处理方式：
   - 从官方接口获取带签名下载链接
   - 把 MSI 做行政解包，拿到本机运行文件
   - 直接启动 `memurai.exe`

实测结果：
- `redis_version: 7.2.4`
- `tcp_port: 6379`
- worker 日志恢复为 `BullMQ workers are running.`

另外，为避免把本机运行文件误推到仓库，本轮在 `.gitignore` 中补了：
- `custom/runtime/`
- `memurai-log.txt`

## 7. 本轮优点

相比上一轮，本轮带来的提升有：

1. 首页人工接管排序从“感性优先级”升级到“有评分依据的稳定排序”。
2. 人工完成开始沉淀结构化结果标签，为后续学习反馈中心打基础。
3. 规则日志从顶层键级推进到深层路径级，更适合回溯复杂规则修改。
4. 本机 Redis 恢复到 BullMQ 可接受版本，自动链路重新具备运行条件。

改善的质量属性包括：
- 可维护性
- 可追踪性
- 运营可读性
- 本机运行稳定性

## 8. 本轮缺点与代价

这一轮也有明确代价和边界：

1. 首页评分结构仍是规则驱动，不是完整机器学习或统计模型。
2. 人工完成结果标签目前是固定枚举，后续可能还要扩容。
3. `ruleContentJson` 现在是路径级摘要，但还不是完整深层结构可视 diff。
4. Redis 目前采用“本机运行文件 + 后台进程”方案，虽然可用，但尚未完全服务化到系统级自启动。
5. 本机 PostgreSQL 当前仍未恢复，`npm run dev` 可启动，但涉及数据库读写的真实业务路径仍依赖数据库环境补齐。

## 10. 本轮结论

本轮最重要的结论是：

第二阶段 P0 现在又往“更像真正的值班操作系统”推进了一步，尤其体现在两条线上：

1. 人工接管开始有更稳定的优先级和结构化收口结果。
2. 自动链的本机 Redis 环境重新回到可运行状态，不再被旧版 Redis 卡住。

被再次证实的经验包括：
- 只做“看起来有优先级”不够，必须把排序结构真正下沉到数据层。
- 只写自由文本说明不够，异常收口一定要补结构化结果标签。
- 端口监听不等于队列可用，BullMQ 版本要求必须真实满足。
