# 版本记录：v0.20-dashboard-owner-filter-manual-history-rule-json-summary

## 1. 本轮目标

本轮目标是在 v0.19 的基础上，把“人工接管”这条线继续往更贴近日常值班的方向补齐，重点解决三个问题：

1. 工作台首页的人工接管提醒还缺少“只看我的”视角，小团队高频使用时仍然要人工筛选。
2. 人工处理完成后虽然已经能填写处理说明，但异常中心还缺少回看这些说明的入口。
3. 规则日志中的 `ruleContentJson` 仍然过于接近原始 JSON，不够适合运营人员快速扫读。

本轮属于：
- 功能补充
- 可维护性提升
- 值班体验增强

## 2. 背景与上下文

本轮承接：
- `custom/notes/v0.19-manual-note-dashboard-links-rule-value-diff.md`
- `custom/notes/v0.18-manual-complete-dashboard-hand-off-rule-diff.md`

修改前的状态是：
- 工作台首页已经能看到人工接管提醒，也能直接跳转处理。
- 异常人工完成时已经支持填写处理说明，并写回异常详情。
- 规则日志已经支持字段级和值级摘要。

但实际使用视角里仍有三个不足：
- 首页提醒还是“所有人混在一起看”，不够贴合 2-3 人小团队轮流处理的习惯。
- 人工处理说明已经写进系统，但异常中心还没有专门把这些说明展示出来。
- `ruleContentJson` 的值级展示仍偏技术化，JSON 一长就影响扫读效率。

## 3. 问题分析

表面看起来，这些都只是展示层的小优化；但真正的核心矛盾是：

1. 系统已经开始具备任务、异常、规则的底盘能力，但值班入口还没有完全贴合“先看我该处理什么”的高频协作场景。
2. 人工接管正在成为第二阶段的重要兜底机制，如果处理说明不能顺手回看，就很难形成真正可复盘的操作链。
3. 规则日志如果对非技术用户仍然偏难读，就很难承担“规则可配置、可追踪、可学习”的系统责任。

主因不是功能缺失，而是信息组织方式还不够“运营化”。

## 4. 候选方案比较

### 方案 A：继续保持现有展示，先把更多新模块做出来

核心思路：
- 不调整首页人工接管提醒
- 不在异常中心补人工完成回看区
- 规则日志继续展示原始 JSON 值

优点：
- 改动最小
- 不增加页面复杂度

缺点：
- 现有值班体验提升有限
- 已经写入的人工说明没有真正发挥作用
- 规则日志仍然不够友好

放弃原因：
- 这会让第二阶段继续偏“有功能但不好用”，不符合当前“质量优先、少跳转、少思考负担”的方向。

### 方案 B：围绕人工接管和规则日志继续做轻量运营化增强

核心思路：
- 首页人工接管提醒增加“只看我的”
- 异常中心新增“最近人工完成”区，专门展示处理说明
- `ruleContentJson` 做人话摘要，不直接堆长 JSON

优点：
- 改动集中在已有页面
- 不需要再扩 schema
- 直接提升小团队值班效率

缺点：
- 首页和异常中心的视图逻辑会比之前多一层
- `ruleContentJson` 仍然只是摘要，不是完整 diff

最终选择原因：
- 这是当前最小、最稳、同时最能体现第二阶段“运营底盘”价值的一组改动。

## 5. 最终决策

本轮最终采用方案 B。

关键决策如下：

1. “只看我的”先做成工作台里的本地过滤，不额外扩接口和查询参数。
2. 人工完成说明先通过异常中心里的“最近人工完成”区回看，优先满足复盘和交接。
3. `ruleContentJson` 的可读性优先靠“键数量 + 关键键预览 + 示例片段”来解决，而不是一上来做复杂 JSON diff 组件。

## 6. 具体实现

本轮修改文件如下：

- `src/app/(dashboard)/dashboard/page.tsx`
- `src/app/(dashboard)/dashboard/page-client.tsx`
- `src/app/(dashboard)/ops/exceptions/page.tsx`
- `src/app/(dashboard)/ops/rules/page-client.tsx`
- `custom/notes/v0.20-dashboard-owner-filter-manual-history-rule-json-summary.md`
- `AGENTS.md`

各文件作用如下：

### 1. `src/app/(dashboard)/dashboard/page.tsx`

重写首页服务端聚合逻辑，补充：
- 当前登录用户读取
- 人工接管提醒所需的 `resolvedById`
- 人工处理说明提取

同时把原来首页里的人工接管展示改为交给客户端面板处理，为“只看我的”提供数据基础。

### 2. `src/app/(dashboard)/dashboard/page-client.tsx`

新增 `DashboardManualExceptionsPanel`，负责：
- 默认只看当前登录用户负责的人工接管异常
- 支持切换“只看我的 / 看全部”
- 在卡片里展示已有处理说明
- 保留最短处理入口

这样首页更贴近真实值班入口，而不是简单信息堆叠。

### 3. `src/app/(dashboard)/ops/exceptions/page.tsx`

重写异常中心页面，补了两层能力：

- 人工接管队列里，如果已经有处理中说明，会直接显示出来
- 新增“最近人工完成”区，展示：
  - 异常类型
  - 处理人
  - 完成时间
  - 处理说明

这样人工说明不再只是埋在 `detailJson` 里，而是能真正参与复盘。

### 4. `src/app/(dashboard)/ops/rules/page-client.tsx`

重点优化 `humanizeValue` 中对 `ruleContentJson` 的处理：
- 对象时显示键数量
- 展示前几个关键键
- 提供简短示例片段
- 避免整段原始 JSON 直接压到日志上

这让规则日志更适合运营人员快速阅读。

## 7. 本轮优点

相比上一轮，本轮的主要提升有：

1. 首页人工接管提醒真正开始贴合“小团队轮值处理”的使用方式。
2. 人工处理说明从“能记录”升级到了“能回看、能复盘”。
3. 规则日志继续向“人话可读”推进，非技术人员理解成本更低。

改善的质量属性包括：
- 可追踪性
- 可复盘性
- 值班效率
- 运营可读性

## 8. 本轮缺点与代价

这轮也有明确边界：

1. “只看我的”目前是前端本地过滤，不是更完整的服务端筛选。
2. “最近人工完成”目前只展示最近若干条，暂时还不是完整历史页。
3. `ruleContentJson` 只是做了友好摘要，还没有做到字段级 JSON 结构 diff。
4. 本轮为了提高稳定性，重写了首页和异常中心页面文件，虽然职责更清楚，但一次性改动量比点改更大。

## 10. 本轮结论

本轮最重要的结论是：

第二阶段 P0 的“人工接管”链路开始真正具备“值班入口 + 处理中上下文 + 处理后复盘”的连续体验了，而不是只有单点动作。

再次被证实的判断包括：
- 小团队协作里，“只看我现在该处理什么”比“看一个大列表”更重要。
- 人工说明只要能在合适的位置回看，就会显著提升异常处理链的价值。
- 规则日志继续往“人话 + 摘要”方向推进，是值得持续坚持的。

## 11. 下一轮建议

下一步最值得继续做的 1~3 件事：

1. 给异常中心补“人工完成后直达相关内容 / 任务”的入口。
2. 给首页人工接管提醒增加更明确的优先级排序。
3. 给规则日志里的 `ruleContentJson` 继续补更细粒度的键级变更摘要。
