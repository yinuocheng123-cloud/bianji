# 版本记录：v0.28-company-review-categories-and-proof-view

## 1. 本轮目标

本轮目标承接 `v0.27-company-feedback-and-evidence-preview`，继续把企业资料自动检索链路从“能提交审核”推进到“能更稳定地审核、能更明确地复盘、能更清楚地识别真实 AI 是否已联调”。

这一轮优先处理三件事：

1. 把企业资料驳回原因从自由文本继续收束成可统计分类。
2. 把企业资料页升级成更强的“字段与证据对照”审核视图，减少来回跳转。
3. 继续核实 OAuth 登录与真实 AI 企业资料检索联调状态，不假装已经完成真实 OpenAI 联调。

本轮属于：

- 功能补充
- 运营体验优化
- 反馈闭环增强

## 2. 背景与上下文

本轮承接：

- `custom/notes/v0.27-company-feedback-and-evidence-preview.md`
- `custom/notes/v0.26-company-discovery-review-flow.md`

上一轮已经完成了：

- AI 企业资料自动检索并提交待审核
- 企业资料页内来源预览和官网证据预览
- 学习反馈中心里的 AI 企业资料通过率 / 驳回率基础统计
- 页面顶部明确显示当前是否配置了 `OPENAI_API_KEY`

但从真实使用上看，仍有三个明显短板：

1. 企业资料驳回仍主要依赖自由文本，后续难以统计、难以反哺规则。
2. 企业资料页虽然能看来源和官网证据，但缺少“字段和证据如何对照”的工作视图。
3. 用户明确提出“使用 OAuth 登录，继续做真实 AI 企业资料检索联调”，因此必须明确区分：
   - OAuth 当前是否可用
   - 企业资料检索当前是否真的已进入真实 OpenAI 联调

## 3. 问题分析

表面现象是：

1. 编辑或管理员能驳回企业资料，但不同人写出的原因口径不一致。
2. 审核企业资料时，要自己在脑中做“字段有没有证据支持”的映射。
3. 页面上已经有 AI 检索入口，容易让人误以为当前已经走通真实大模型链路。

真正的核心矛盾是：

- 第二阶段强调“系统持续积累经验”，但如果驳回原因不结构化，企业资料链路就无法稳定学习。
- 小团队高频审核时，最大的成本不是没有按钮，而是要自己在来源、官网候选和字段之间反复切换。
- 真实 AI 联调属于高风险事实判断；没有 `OPENAI_API_KEY` 时，必须诚实停留在检索兜底模式。

## 4. 候选方案比较

### 方案 A：只补驳回分类，不动企业页对照视图

核心思路：

- 给驳回接口增加分类字段，先让反馈中心有统计数据。

优点：

- 改动小。
- 能快速让反馈中心出统计。

缺点：

- 企业资料页审核体验几乎不变。
- “分类统计”和“审核动作”之间仍然断开。

放弃原因：

- 这轮目标不只是让后台多一个统计数字，而是要让审核效率一起提升。

### 方案 B：同时补驳回分类、反馈统计、字段与证据对照视图

核心思路：

- 在 `CompanyProfile` 上正式记录驳回分类。
- 驳回接口强制带分类。
- 反馈中心统计 AI 企业资料驳回分类。
- 企业资料页把字段清单、来源证据、官网候选证据和审核提示集中显示。

优点：

- 结构化反馈和审核体验同步提升。
- 不新增独立审核系统，仍然复用现有企业资料页。
- 更符合当前小团队少跳转的使用方式。

缺点：

- 企业资料客户端面板会更复杂。
- 需要同步调整 schema、接口、反馈中心和页面。

最终选择原因：

- 这是当前最接近“可持续运营”的增量方案，而且仍然控制在现有企业资料模块里。

## 5. 最终决策

本轮最终采用方案 B，关键决策如下：

1. 新增 `CompanyReviewIssueCategory` 枚举，并写入 `CompanyProfile.reviewIssueCategory`。
2. 企业资料驳回时不再只收自由文本，同时要求填写驳回分类。
3. 企业资料页在现有来源预览、官网证据预览之外，再增加“字段与证据对照”工作区。
4. 学习反馈中心新增 AI 企业资料驳回分类统计，形成可复盘口径。
5. OAuth 登录继续沿用当前 Google dev bypass 验证链路；真实 OpenAI 联调以 `.env.local` 中 `OPENAI_API_KEY` 是否存在为准，不伪造结果。

## 6. 具体实现

本轮改动文件如下：

- `prisma/schema.prisma`
- `prisma/seed.mjs`
- `src/lib/constants.ts`
- `src/app/api/companies/[id]/reject/route.ts`
- `src/app/api/companies/[id]/approve/route.ts`
- `src/app/api/companies/[id]/route.ts`
- `src/app/(dashboard)/companies/page.tsx`
- `src/app/(dashboard)/companies/page-client.tsx`
- `src/app/(dashboard)/ops/feedback/page.tsx`
- `custom/notes/v0.28-company-review-categories-and-proof-view.md`
- `AGENTS.md`

### 1. Schema 与数据口径

在 `prisma/schema.prisma` 中：

- 新增 `CompanyReviewIssueCategory`
- `CompanyProfile` 新增 `reviewIssueCategory`

这样企业资料驳回原因从自由文本升级成了可统计的结构化字段。

### 2. 企业资料驳回接口

在 `src/app/api/companies/[id]/reject/route.ts` 中：

- 驳回接口新增 `category`
- 只允许以下分类：
  - `SOURCE_INSUFFICIENT`
  - `WEBSITE_EVIDENCE_INSUFFICIENT`
  - `MISSING_FIELDS`
  - `CONFLICT_IDENTIFICATION`
  - `OTHER`

在 `src/app/api/companies/[id]/approve/route.ts` 中：

- 审核通过时清空 `reviewIssueCategory`

### 3. 企业资料页审核视图增强

在 `src/app/(dashboard)/companies/page-client.tsx` 中：

- 驳回动作新增分类输入
- 保留来源预览
- 保留官网证据预览
- 新增“字段对照”工作区，集中展示：
  - 企业名称
  - 品牌名称
  - 地区
  - 官网
  - 定位
  - 主营产品
  - 当前审核结论
  - 审核提示
  - 主要来源证据
  - 官网候选证据

这里的关键取舍是：

- 没有另起详情页
- 没有继续拆更多弹层
- 而是把审核时最常来回比对的信息尽量压到同一块工作区

### 4. 反馈中心继续接企业资料驳回分类

在 `src/app/(dashboard)/ops/feedback/page.tsx` 中：

- 新增 AI 企业资料驳回分类统计
- 最近 AI 企业资料审核样本中，回显驳回分类 badge
- 保留已有：
  - 审核通过 / 驳回 / 待修订统计
  - 人工接管结果标签
  - 高频驳回原因
  - 来源质量排行

### 5. 企业资料页面服务端查询修正

在 `src/app/(dashboard)/companies/page.tsx` 中：

- 统一改为 `select`
- 明确查询 `reviewIssueCategory`
- 避免把标量字段写进 `include` 的不稳定写法继续扩散

### 6. 种子数据补驳回分类样本

在 `prisma/seed.mjs` 中：

- 给已驳回的 AI 企业资料样本补了 `SOURCE_INSUFFICIENT`

另外本轮还顺手修正了一个兼容性问题：

- `seed.mjs` 继续使用字符串字面量写入枚举值，避免 ESM 直接从 `@prisma/client` 命名导入新枚举时的 CommonJS 边界问题。

## 7. 本轮优点

本轮带来的主要改进有：

1. 企业资料驳回第一次真正进入“可统计、可复盘、可反馈”的状态。
2. 企业资料审核从“看几个预览卡片”升级成“字段和证据对照”的工作视图。
3. 反馈中心第一次能直接回答：AI 企业资料主要是因为什么被驳回。
4. OAuth 登录链路仍可用，企业资料 AI 检索链路仍可运行，不影响当前团队继续测试。

改善的质量属性包括：

- 可运营性
- 可复盘性
- 审核效率
- 可维护性

## 8. 本轮缺点与代价

这一轮也有明确代价：

1. 企业资料客户端页进一步变大，后续可能需要拆成更细的子组件。
2. 驳回分类目前仍由用户通过 prompt 输入，不是更完整的固定弹层表单。
3. “字段与证据对照”目前是面向当前字段的工作视图，还不是更重的结构化证据评审系统。
4. 真实 OpenAI 联调仍未完成，因为当前 `.env.local` 中 `OPENAI_API_KEY=""`。

## 9. 验证与结果

本轮执行了以下验证：

- `git fetch --all --prune`
- `git pull --ff-only`
- `npm run db:generate`
- `npx prisma db push`
- `node prisma/seed.mjs`
- `npm run lint`
- `npm run build`
- 访问：
  - `/login`
  - `/companies`
  - `/ops/feedback`
  - `/api/auth/google/start`

验证结果：

1. `db push` 成功，数据库 schema 已同步。
2. `seed` 成功。
3. `lint` 通过。
4. `build` 通过。
5. `/login` 返回 `200`。
6. `/companies` 返回 `200`。
7. `/ops/feedback` 返回 `200`。
8. `/api/auth/google/start` 返回 `307`，说明 OAuth 发起链路仍然存在。

需要单独说明的点：

- `npm run db:generate` / `db push` 里的 generate 阶段仍出现 Prisma Windows 引擎文件重命名 `EPERM`，这是当前本机开发进程占用导致的已知环境问题；本轮 schema 同步和构建验证不受影响。
- 当前 `.env.local` 中 `OPENAI_API_KEY=""`，因此真实 OpenAI 企业资料检索联调仍然阻塞，没有发生实际大模型调用。

## 10. 本轮结论

本轮最重要的结论是：

企业资料自动检索链路现在已经从“能提交审核”继续推进到“能更稳定地审核、能更明确地复盘”。

被证实的判断包括：

1. 驳回分类必须结构化，否则企业资料链路无法稳定学习。
2. 对当前小团队来说，“字段与证据对照”比继续拆新页面更有直接收益。
3. OAuth 登录可以继续沿用当前 dev bypass 作为本机验证手段。
4. 真实 OpenAI 联调不能靠猜测，必须以真实密钥存在与否为准。

## 11. 下一轮建议

下一轮最值得继续做的 1~3 件事：

1. 把企业资料驳回分类继续细化到字段级缺失统计。
2. 把企业资料审核弹层从 prompt 升级成固定表单弹层。
3. 在你提供有效 `OPENAI_API_KEY` 后，继续做真实 OpenAI 企业资料检索联调，并记录真实输出质量。
