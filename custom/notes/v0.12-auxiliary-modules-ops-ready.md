# 版本记录：v0.12-auxiliary-modules-ops-ready

## 1. 本轮目标

本轮目标是把辅助模块从“能展示数据”推进到“可执行运营维护”。

这轮目标属于：
- 功能补充
- 可维护性提升

优先处理这个问题，是因为三步工作台和主流程虽然已经可用，但辅助模块仍停留在只读列表阶段，实际运营时无法直接维护关键词、站点、企业资料和提示词，也不方便做日志检索和抽取结果追踪。

## 2. 背景与上下文

本轮承接：
- `custom/notes/v0.11-dev-oauth-bypass.md`
- `custom/notes/v0.10-content-detail-workbench.md`

修改前状态：
- 关键词、站点、企业资料、提示词页面基本是只读表格
- 对应后端 API 大多已经具备新增或更新能力
- 日志和抽取结果页面以展示为主，缺少检索和直达入口
- 企业资料和提示词还缺删除闭环

因此这轮更适合围绕现有 API 补运营面板，而不是重做后端结构。

## 3. 问题分析

表面问题：
- 用户能看到辅助模块，但很难直接拿来做日常维护
- 页面缺少搜索、筛选、创建、编辑、启停等高频动作
- 日志和抽取结果不利于运营排查

核心矛盾：
- 后端能力已经有了一部分
- 前端仍然停留在 MVP 展示层

主要原因：
1. 之前优先保证了主流程跑通
2. 辅助模块为了尽快有骨架，先只做了服务端列表渲染
3. 部分删除和更新能力没有完整闭环

## 4. 候选方案比较

### 方案 A：继续维持只读列表，依赖 API 手工调用

核心思路：
- 页面继续只做展示
- 需要维护时由开发者手工调接口

优点：
- 前端改动最少

缺点：
- 不适合运营使用
- 用户无法在页面上直接执行维护

放弃原因：
- 与“到可执行运营层面”的目标冲突

### 方案 B：在现有字段和 API 基础上补运营面板

核心思路：
- 关键词、站点、企业资料、提示词补搜索 + 创建/编辑 + 状态维护
- 日志补搜索、模块筛选和详情展开
- 抽取结果补搜索和直达入口
- 缺失的删除/来源更新能力在 API 层补齐

优点：
- 不改 schema
- 与现有后端结构兼容
- 能明显提升日常可用性

缺点：
- 新增多个客户端页面文件
- 部分模块仍然是轻量操作台，不是完整复杂后台

最终选择原因：
- 这是最符合当前项目阶段的增量方案

## 5. 最终决策

本轮采用方案 B。

关键决策如下：

1. 不动数据库结构
2. 优先把高频运营动作接到现有 API 上
3. 对风险较高的删除操作只补在企业资料和提示词上
4. 关键词和站点优先用“启停”代替激进删除

## 7. 本轮优点

当前方案相比之前更好的地方：

- 辅助模块不再只是看板，而是开始具备运营动作
- 后台已经形成“主流程 + 辅助维护”双层结构
- 日常高频维护不再需要脱离页面手工调接口
- 日志和抽取结果开始具备排查价值

得到改善的质量属性：
- 可用性
- 可维护性
- 运营可执行性

## 8. 本轮缺点与代价

当前方案的已知限制：

1. 这些面板仍然是轻量运营台，不是复杂表单系统
2. 站点和关键词没有补删除，而是偏向启停管理
3. 企业资料中的人物信息仍使用轻量文本列表，而不是复杂结构化编辑器
4. 提示词模板还没有版本历史

这些代价当前是可接受的，因为这轮目标是先把辅助模块推到“可执行运营层面”，而不是一步做成重量级后台。
