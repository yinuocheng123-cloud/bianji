# 版本记录：v0.24-postgres-feedback-grouped-rule-diff

## 1. 本轮目标

本轮目标有三件事，而且三件事必须一起完成才算收口：

1. 恢复本机 PostgreSQL，让真实数据库链路重新跑通，而不是继续停留在“页面能开、数据库不可用”的半可用状态。
2. 把人工完成结果标签正式接进学习反馈中心统计，开始把异常收口结果转成后续可复用的运营信号。
3. 继续把 `ruleContentJson` 的路径摘要升级成更清晰的分组 diff，降低规则变更复盘成本。

本轮属于：

- 运行环境修复
- 功能补充
- 可维护性提升

## 2. 背景与上下文

本轮承接：

- `custom/notes/v0.23-manual-scoring-tags-deep-diff-redis.md`
- `custom/notes/v0.22-start-work-button-motion.md`
- `custom/notes/v0.21-manual-links-priority-rule-key-diff.md`

上一轮已经完成了：

- 本机 Redis 恢复到 BullMQ 可接受版本
- 人工接管优先级评分
- 人工完成结果标签
- `ruleContentJson` 的路径级摘要

但还留下了三个明显短板：

1. PostgreSQL 仍未恢复，导致真实业务流虽然界面可访问，但数据库读写并不稳定。
2. 学习反馈中心仍是占位页，人工完成结果标签还没有正式进入统计口径。
3. 规则日志虽然能看到路径变化摘要，但还不够适合非技术角色快速看清“哪一组规则变了什么”。

## 3. 问题分析

表面现象分别是：

1. 本机 `localhost:5432` 不可用，登录、列表、种子数据和真实页面都缺少稳定数据库支撑。
2. 人工完成动作已经有结构化标签，但没有在反馈中心沉淀，等于写了日志却没有形成学习视图。
3. 规则日志只给路径摘要，不做分组，导致复杂规则一旦嵌套较深，还是很难快速扫读。

真正的核心矛盾是：

- 第二阶段已经在做“操作系统化”，不能只把界面做出来而让运行底盘继续虚空。
- 人工处理结果如果不能进入反馈中心，就无法反哺规则和模板。
- 规则变更如果看不清，就很难支撑 2-3 人团队的高频协作和复盘。

## 4. 候选方案比较

### 方案 A：只修页面，不动本机数据库环境

核心思路：

- 继续保留 `withFallback` 兜底
- 只补反馈中心页面和规则 diff 展示
- PostgreSQL 后面再单独处理

优点：

- 代码改动最少
- 本轮推进快

缺点：

- 真实业务流仍然不稳定
- 页面统计可能只是兜底数据，不具备运营价值
- 和本轮用户目标不一致

放弃原因：

- 用户明确要求“先恢复本机 PostgreSQL，把真实业务流一起跑通”，不能跳过。

### 方案 B：先恢复 PostgreSQL，再补反馈中心与分组 diff

核心思路：

- 先用本机可携带 PostgreSQL 二进制恢复 `5432`
- 再执行 `db push` 和种子数据写入
- 最后补反馈中心和规则 diff，让页面直接建立在真实数据库之上

优点：

- 真实业务流能一起恢复
- 反馈中心统计有真实数据承接
- 规则 diff 的验证也更可靠

缺点：

- 本轮会同时涉及运行环境和页面逻辑
- Windows 下本机 PostgreSQL 处理比普通页面修改更繁琐

最终选择原因：

- 这是唯一真正满足本轮要求的方案。

## 5. 最终决策

本轮最终采用方案 B，关键决策如下：

1. PostgreSQL 不再继续依赖损坏的半下载压缩包，而是先确认官方包完整性，再只提取 `pgsql/` 目录用于本机运行。
2. 数据库恢复后，不额外扩 schema，学习反馈中心先基于现有 `ReviewAction` 和 `ExceptionEvent` 聚合出第一版正式统计。
3. 规则 diff 不做整页重构，而是在现有路径展开能力上继续加一层“按顶层组分组”的可视化展示。
4. 为了让反馈中心立刻可见，种子数据补充了审核动作和带结果标签的人工收口异常。

## 6. 具体实现

本轮改动文件如下：

- `src/app/(dashboard)/ops/feedback/page.tsx`
- `src/app/(dashboard)/ops/rules/page-client.tsx`
- `prisma/seed.mjs`
- `eslint.config.mjs`
- `tsconfig.json`
- `custom/notes/v0.24-postgres-feedback-grouped-rule-diff.md`
- `AGENTS.md`

### 1. PostgreSQL 本机恢复

运行环境层面做了这些动作：

- 重新确认官方 PostgreSQL 16.13 Windows x64 二进制包完整长度。
- 继续断点续传下载，补齐损坏的 ZIP 结尾。
- 只提取 `pgsql/` 目录到 `custom/runtime/postgresql17/`，避免再次被 `pgAdmin` 长路径干扰。
- 使用 `initdb` 初始化数据目录到 `custom/runtime/postgresql17-data/`。
- 使用 `pg_ctl` 启动本机 PostgreSQL，监听 `127.0.0.1:5432` 和 `::1:5432`。
- 创建数据库 `editorial`。

### 2. 学习反馈中心正式接入统计

在 `src/app/(dashboard)/ops/feedback/page.tsx` 中：

- 用 `ReviewAction` 聚合：
  - 审核通过数
  - 驳回数
  - 待修订数
  - 总审核动作数
- 用 `ReviewAction` 近 7 天数据生成质量趋势
- 用 `ExceptionEvent.detailJson.manualResolutionTag` 聚合人工完成结果标签统计
- 增加：
  - 高频驳回原因
  - 最近人工收口样本
  - 最近人工完成标签分布

这样反馈中心不再是占位页，而是第一版正式统计页。

### 3. 规则 diff 升级为分组展示

在 `src/app/(dashboard)/ops/rules/page-client.tsx` 中：

- 保留原有路径拍平逻辑
- 新增路径分组结构：
  - `RuleContentPathChange`
  - `RuleContentDiffGroup`
- 以路径首段作为分组键
- 在日志渲染中展示：
  - 组标签
  - 新增 / 删除 / 变更
  - 路径
  - before -> after 预览值

这一步把“路径摘要”进一步升级成了更适合扫读的“分组 diff”。

### 4. 种子数据补齐反馈样本

在 `prisma/seed.mjs` 中：

- 增加一条额外草稿 `seed-draft-002`
- 补三条审核动作：
  - 通过
  - 驳回
  - 需修订
- 补一条带 `manualResolutionTag=FIXED_DATA` 的人工完成异常

这样反馈中心和异常反馈不再只显示空数据。

### 5. 运行文件排除构建污染

在 `eslint.config.mjs` 和 `tsconfig.json` 中：

- 显式排除 `custom/runtime/**`
- 避免 PostgreSQL 附带的 `pgAdmin 4` 前端源码被 `eslint` 和 `next build` 误扫进项目

这样既保留了本机运行文件，又不会破坏正常构建。

## 7. 本轮优点

相比之前，本轮的主要提升有：

1. 本机数据库真正恢复，`localhost:5432` 回到可用状态，真实业务流不再悬空。
2. 学习反馈中心从占位页升级为正式统计页，人工完成结果标签开始进入系统反馈口径。
3. 规则日志的 `ruleContentJson` 展示更接近运营视角，而不是只给开发者看路径字符串。
4. 种子数据更完整，首次打开反馈中心就能看到可解释的样本。

改善的质量属性包括：

- 运行稳定性
- 可追踪性
- 运营可读性
- 后续可学习性

## 8. 本轮缺点与代价

本轮也有明确代价和边界：

1. PostgreSQL 目前采用本机可携带运行方式，已经可用，但还不是系统服务化启动。
2. 学习反馈中心目前仍是第一版统计，没有补字段级编辑频率、模板效果和来源质量评分。
3. `ruleContentJson` 分组 diff 是按顶层组分组，不是完整深层树形 diff。
4. `prisma db push` 虽然完成 schema 同步，但在最后的 generate 阶段仍受到当前开发进程对 Prisma 引擎文件的占用影响，因此本轮采用“db push + 直接 seed”的最小可行验证路径。

## 9. 验证与结果

本轮执行了以下验证：

- `git fetch --all --prune`
- `postgres.exe --version`
- `initdb`
- `pg_ctl start`
- `psql -h localhost -U postgres -d postgres -c "select version();"`
- `createdb editorial`
- `npx prisma db push`（带 `DATABASE_URL`）
- `node prisma/seed.mjs`（带 `DATABASE_URL`）
- `npm run lint`
- `npm run build`
- `http://127.0.0.1:3000/login`

验证结果：

1. PostgreSQL 已恢复为本机可运行状态，端口 `5432` 正常监听。
2. `psql` 可连接，数据库版本为 PostgreSQL 16.13。
3. 数据库 `editorial` 已创建。
4. schema 已同步，种子数据已成功写入。
5. 反馈中心与规则页代码已通过 lint 和 build。
6. 开发服务可正常访问登录页。

额外观察：

- `prisma db push` 结束后的 generate 阶段，仍会因为当前开发环境对 Prisma 引擎 DLL 的占用出现 `EPERM rename`。这不影响数据库 schema 同步本身，但后续如果需要重新 generate，最好先停掉占用中的开发进程。

## 10. 本轮结论

本轮最重要的结论是：

第二阶段不能再把“环境恢复”和“反馈沉淀”分开做。只有本机数据库真恢复了，反馈中心和规则 diff 的升级才算真正进入可运营阶段。

被再次证实的经验包括：

- 本机端口能连通才算业务流真正恢复，不是页面能打开就算完成。
- 人工完成结果标签一旦进入反馈中心，就开始具备反哺规则优先级的价值。
- 复杂规则变更必须做运营可读的分组 diff，否则日志只会继续停留在开发自用层。

## 11. 下一轮建议

下一轮最值得继续做的 1~3 件事：

1. 把学习反馈中心继续补上来源质量和模板效果统计。
2. 把人工完成结果标签进一步接入规则优化建议或推荐动作逻辑。
3. 给 PostgreSQL 补一个更稳定的本机启动脚本或服务化启动方案，避免重启机器后再次手动恢复。
