# 版本记录：v0.4-review-comments-and-queue-retry

## 1. 本轮目标

在上一轮已经补齐“工作台可见性”和“审核快捷动作”的基础上，继续推进两件更贴近日常值班的能力：

- 审核快捷动作支持填写审核意见
- 工作台支持看到失败任务并一键重试

这轮目标属于：

- 功能补充
- 可维护性提升
- 协作闭环增强

优先做这轮的原因是，上一轮虽然已经让审核可以更快处理、首页可以看见进度，但仍然缺少两个关键闭环：

1. 快捷审核没有意见输入，处理虽然快，但信息不完整
2. 首页能看到流程进度，却还不能处理失败任务

## 2. 背景与上下文

本轮承接：

- `custom/notes/v0.3-ops-visibility-and-review-actions.md`

上一轮形成的新基线是：

- 首页已有流程总览和近期关注
- 审核修订区已有快捷处理按钮

继续从真实操作链路看，问题已经从“能不能用”转向“出了问题后怎么恢复、人工判断怎样留下明确痕迹”。

## 3. 问题分析

表面现象：

- 快捷通过 / 退回只能走固定文案，审核员没法顺手写下本次意见
- 工作台可以看到总量和进度，但失败任务仍然需要去更底层排查
- Worker 失败没有统一写入工作流日志，异常缺少闭环留痕

核心矛盾：

- 当前主流程已经可用，但异常处理和审核痕迹还不够完整

主要原因：

1. 之前优先补的是主流程效率，不是审核意见完整性
2. 队列只承担了任务执行，没有进一步向工作台暴露健康状态
3. Worker 完成与失败事件没有系统化记录

## 4. 候选方案比较

### 方案 A：只补审核意见，不处理失败任务

核心思路：

- 审核快捷操作里增加意见输入
- 队列与工作台部分不动

优点：

- 改动小
- 风险低

缺点：

- 只能补齐人工审核闭环，仍然没有解决失败任务恢复问题

放弃原因：

- 工作台已经是主入口，不补失败重试会让“值班控制台”的定位不完整

### 方案 B：审核意见输入 + 队列失败重试一起补

核心思路：

- 审核快捷操作支持填写意见
- 队列工具层暴露 waiting / active / failed / delayed
- 新增失败任务重试接口
- Worker 成功/失败写入日志

优点：

- 同时补齐人工审核闭环和自动流程异常恢复
- 继续沿用既有架构，不需要改数据库 schema

缺点：

- 要同时改工作台、队列封装、Worker 和审核组件

风险：

- 如果队列统计类型设计不稳，容易在工作台页面触发类型问题
- 如果重试策略太激进，可能把失败任务重复推进

最终选择原因：

- 这是当前最值得做的一步，能明显提高后台的“值班可用性”

## 5. 最终决策

本轮最终采用方案 B，具体决策如下：

1. `src/lib/queue.ts` 增加统一队列健康状态查询
2. 新增 `POST /api/control/retry-failed`
3. `GET /api/control/status` 返回队列健康信息和各队列明细
4. 工作台客户端增加“异常与重试”面板
5. Worker 在完成和失败时都写日志
6. 审核快捷操作增加意见输入框，再调用既有草稿接口

## 6. 具体实现

本轮修改文件如下：

- `src/lib/queue.ts`
- `src/app/api/control/status/route.ts`
- `src/app/api/control/retry-failed/route.ts`
- `src/workers/start.ts`
- `src/app/(dashboard)/dashboard/page.tsx`
- `src/app/(dashboard)/dashboard/page-client.tsx`
- `src/app/(dashboard)/review/review-actions.tsx`
- `AGENTS.md`

各文件作用：

### `src/lib/queue.ts`

- 新增 `QueueSummary` 类型
- `getManagedQueuesStatus()` 现在除了运行状态，还返回：
  - waiting
  - active
  - failed
  - delayed
- 新增 `retryManagedFailedJobs()`

这里的设计选择是继续把所有队列控制收敛在一个地方，避免工作台和路由层直接操作各条队列细节。

### `src/app/api/control/status/route.ts`

- 在原有待处理数量基础上，额外返回：
  - `queueHealth`
  - `queues`

这样工作台前端不需要再拼接多个接口，直接一次拿到状态。

### `src/app/api/control/retry-failed/route.ts`

- 统一重试三条主流程队列中的失败任务
- 写入操作日志

这里采用“按队列批量重试”的保守方案，而不是自动无限重试。这样仍然保留人工判断入口。

### `src/workers/start.ts`

- 重写 Worker 入口结构
- 在 `completed` 和 `failed` 事件上统一写入日志

这样一来，首页“最近动态”和日志页能真正看到队列事件，而不是只记录人工操作。

### `src/app/(dashboard)/dashboard/page.tsx`

- 对齐新的工作台状态结构
- 把首页服务端传给客户端的 `initialStatus` 增加队列健康信息

### `src/app/(dashboard)/dashboard/page-client.tsx`

- 新增“异常与重试”面板
- 展示：
  - 等待中
  - 处理中
  - 失败任务
  - 延迟中
- 每条队列展示明细
- 新增“重试失败任务”按钮

### `src/app/(dashboard)/review/review-actions.tsx`

- 增加“填写意见 / 收起意见”
- 审核员可以先写意见，再执行通过或退回
- 如果没写意见，仍保留默认文案兜底

## 7. 本轮优点

本轮相比上一轮的提升主要体现在：

- 审核快捷操作不再只有动作，没有说明
- 工作台不再只能看进度，也能处理失败任务
- 队列失败与成功开始进入正式日志链路
- “三步工作台”更像真实值班台，而不是纯展示面板

得到改善的质量属性：

- 可维护性
- 可观察性
- 异常恢复能力
- 协作完整性

## 8. 本轮缺点与代价

本轮仍有这些限制：

1. 失败任务重试目前是手动触发，不是自动重试策略
2. 每条队列重试上限暂时取 50 条，适合当前轻量商用阶段，但大规模场景后续仍要加强
3. 审核意见输入目前是行内展开，不是更完整的弹层体验
4. 失败任务详情还没有直接展示具体报错明细，只能通过日志继续追踪
5. Redis 版本偏低的环境提示仍在

这些代价当前可接受，因为本轮重点是先把异常处理闭环做出来，而不是一次性做成完整运维台。

## 10. 本轮结论

本轮最重要的结论是：

三步工作台已经开始具备“异常可恢复、审核有痕迹”的基本值班能力，而不再只是一个流程启动器。

被证实的判断：

- 只做增量修改，不改数据库结构，也能显著增强异常处理闭环
- 队列健康状态与失败重试很适合继续挂在工作台主入口上

后续应默认继承的经验：

- 自动流程相关增强，优先考虑是否能挂到工作台异常面板
- 审核相关增强，优先考虑是否能在快捷动作里保留足够上下文

## 11. 下一轮建议
