# 版本记录：v0.21-manual-links-priority-rule-key-diff

## 1. 本轮目标

本轮目标是在 v0.20 的基础上，把人工接管和规则日志再往“高频值班可执行”推进一层，围绕三件事收口：

1. 给“最近人工完成”补直达相关内容或任务的入口。
2. 给首页人工接管提醒补更明确的优先级排序和排序依据。
3. 继续细化 `ruleContentJson` 的键级变更摘要。

本轮属于：
- 功能补充
- 可维护性提升
- 运营可读性增强

## 2. 背景与上下文

本轮承接：
- `custom/notes/v0.20-dashboard-owner-filter-manual-history-rule-json-summary.md`
- `custom/notes/v0.19-manual-note-dashboard-links-rule-value-diff.md`

修改前的状态是：
- 首页已经支持“只看我的”人工接管提醒。
- 异常中心已经能回看最近人工完成和处理说明。
- 规则日志已经能对 `ruleContentJson` 做对象级友好摘要。

但离更稳定的运营感还有三个缺口：
- 人工完成回看时还缺少“看完马上去相关对象”的最短入口。
- 首页人工接管虽然能筛选“只看我的”，但还没有把优先级显式表达出来。
- `ruleContentJson` 仍然停留在“对象摘要”，没有进入更明确的键级变化说明。

## 3. 问题分析

这一轮的核心问题已经不是“有没有入口”，而是“入口是否足够短、提示是否足够清楚、日志是否足够可读”。

主要矛盾有三点：

1. 人工完成记录如果只能回看说明，不能继续跳到相关任务或内容，复盘就会断一截。
2. 首页提醒如果没有优先级和依据，小团队虽然能看到列表，但仍然要自己二次判断先做哪条。
3. `ruleContentJson` 如果只展示对象级概括，仍然不够支持“规则到底改了哪些键”的快速理解。

所以这轮的重点不是扩模块，而是继续压缩思考负担和跳转成本。

## 4. 候选方案比较

### 方案 A：维持现有提醒和日志，只加少量按钮

核心思路：
- 给最近人工完成加一个通用“回异常中心”按钮
- 首页仍按时间排序
- 规则日志继续只做对象级摘要

优点：
- 改动最小
- 页面结构几乎不变

缺点：
- 人工完成仍然不能真正回到处理对象
- 首页优先级仍然靠人脑判断
- 规则日志对 `ruleContentJson` 的价值提升有限

放弃原因：
- 这不能实质性提升值班效率，只是多了按钮，不符合这轮目标。

### 方案 B：补最短入口、显式优先级、键级变化摘要

核心思路：
- 用统一映射把异常指回内容工作区或任务详情
- 首页人工接管按“是否分配给我 + 优先级 + 更新时间”排序
- `ruleContentJson` 增加“新增键 / 删除键 / 变更键”摘要

优点：
- 更贴近真实日常处理顺序
- 直接减少判断成本
- 规则日志更适合运营和管理者快速扫读

缺点：
- 页面和客户端逻辑会再复杂一点
- `ruleContentJson` 仍然不是完整深层 diff

最终选择原因：
- 这是当前最小代价下，最能提升“商用值班台感”的方案。

## 5. 最终决策

本轮最终采用方案 B。

关键决策如下：

1. 最近人工完成直接复用异常到对象的映射逻辑，不单独发明新跳转规则。
2. 首页人工接管优先级采用轻量规则：
   - 核心异常类型优先
   - 与任务直接相关或已积压较久的次之
   - 其余排队处理
3. `ruleContentJson` 先做顶层键级 diff 摘要，不扩成更复杂的深层结构比较。

## 6. 具体实现

本轮修改文件如下：

- `src/app/(dashboard)/dashboard/page.tsx`
- `src/app/(dashboard)/dashboard/page-client.tsx`
- `src/app/(dashboard)/ops/exceptions/page.tsx`
- `src/app/(dashboard)/ops/rules/page-client.tsx`
- `custom/notes/v0.21-manual-links-priority-rule-key-diff.md`
- `AGENTS.md`

### 1. `src/app/(dashboard)/dashboard/page.tsx`

补充人工接管优先级计算：
- `priority`
- `priorityLabel`
- `priorityReason`

优先级依据会随着数据一起下发到首页面板，不把判断埋在客户端里。

### 2. `src/app/(dashboard)/dashboard/page-client.tsx`

重写了人工接管提醒面板：
- 默认仍支持“只看我的”
- 现在会按“是否分配给我 -> 优先级 -> 更新时间”排序
- 卡片会显示优先级标签和优先级依据

这样首页不只是知道“有几条异常”，而是更清楚“先做哪条、为什么先做”。

### 3. `src/app/(dashboard)/ops/exceptions/page.tsx`

补了异常目标映射 `getExceptionTargetLink`：
- 内容异常优先回内容工作区
- 任务异常优先回任务详情
- 其余回异常中心

并在两个区域接入：
- 人工接管队列
- 最近人工完成

现在人工完成记录不只是“看说明”，还能直接回到相关对象继续处理或复盘。

### 4. `src/app/(dashboard)/ops/rules/page-client.tsx`

重写了规则日志展示逻辑，并新增：
- `getRuleContentKeyDiff`
- `summarizeRuleContentDiff`

现在 `ruleContentJson` 除了对象级摘要，还会显示：
- 新增键
- 删除键
- 变更键

这让规则内容变化更适合日常扫读和追踪。

## 7. 本轮优点

本轮相比上一轮的主要提升有：

1. 人工完成不再停留在“有记录”，而是能继续跳回相关对象。
2. 首页人工接管提醒开始具备更明确的处理顺序。
3. 规则日志对 `ruleContentJson` 的表达从对象级提升到了顶层键级。

改善的质量属性包括：
- 值班效率
- 可追踪性
- 复盘便利性
- 运营可读性

## 8. 本轮缺点与代价

这一轮的代价和边界也很明确：

1. 首页优先级规则目前仍是轻量启发式，不是完整评分系统。
2. 最近人工完成的直达入口仍然依赖异常上下文是否足够完整。
3. `ruleContentJson` 现在是顶层键级摘要，还不是深层对象级 diff。
4. 为了提高稳定性，本轮继续重写了两个客户端页面文件，改动量相对大于点改。

## 10. 本轮结论

本轮最重要的结论是：

第二阶段 P0 的人工接管链路，已经开始从“能处理”进入“更像正式值班系统”的阶段了。

被再次证实的判断包括：
- 首页提醒必须同时给出优先级和理由，才真正能减少思考负担。
- 人工完成记录只有能继续回到对象，才算真正闭环。
- 规则日志里对 `ruleContentJson` 的键级表达，确实比纯对象摘要更接近实际运营需要。

## 11. 下一轮建议

下一步最值得继续做的 1~3 件事：

1. 把人工接管优先级从启发式规则升级为更明确的评分结构。
2. 给最近人工完成补“处理结果标签”而不只是文本说明。
3. 继续细化 `ruleContentJson` 的深层键变化，尤其是嵌套对象和数组。
