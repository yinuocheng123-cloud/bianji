# 版本记录：v0.30-commercial-readiness-ui-runtime-hardening

## 1. 本轮目标

本轮目标是对“接口之外”的高频问题做一轮商用化修补，重点不再扩功能，而是补运行稳定性、审核体验和构建稳定性。

这一轮优先级高，原因有三点：

1. 现有系统虽然功能链路已经比较完整，但仍有明显“开发态痕迹”，例如 `prompt/confirm`、远程字体依赖和本机脚本不稳。
2. 这些问题不会直接表现为接口报错，却会持续消耗编辑和管理员的日常使用体验。
3. 第二阶段要继续往“操作系统化”推进，前端交互和本机运行底盘必须先收稳，否则后续自动链、AI 联调和审核体验都会受影响。

本轮属于：

- 可维护性提升
- 稳定性增强
- 使用体验升级

## 2. 背景与上下文

本轮承接：

- `custom/notes/v0.29-ai-provider-layer-and-openai-config.md`
- `custom/notes/v0.28-company-review-categories-and-proof-view.md`

修改前的状态是：

1. `src/app/layout.tsx` 仍依赖 `next/font/google`，导致构建对外网 Google Fonts 资源有依赖。
2. 企业资料页已经承担了“AI 提交待审核”的核心角色，但审核和删除仍然依赖 `window.prompt` / `window.confirm`。
3. 本机 PostgreSQL 和 Redis 虽然可手工恢复，但缺少稳定、可重复的脚本闭环。
4. Redis 运行态并不稳定，导致构建和任务队列经常出现 `ECONNREFUSED` 噪音。

已知约束：

- 本轮不继续改接口协议，不重做数据库设计。
- 继续沿用现有 Next.js + Prisma + PostgreSQL + Redis/BullMQ 架构。
- 必须保持三步工作台、企业资料审核和 AI Provider 层现有能力不回退。

## 3. 问题分析

表面问题是：

1. 页面能用，但不像正式后台。
2. 本机能跑，但恢复依赖经验，不像可运维环境。
3. build 虽然能过，但存在对网络和本地服务状态的隐性依赖。

真正的核心矛盾是：

- 系统已经进入“持续运营”阶段，影响体验和稳定性的细节问题，价值已经高于继续堆新页面。

当前主要瓶颈：

1. 远程字体依赖让构建稳定性受外网影响。
2. 审核页交互过于临时，不适合管理人员高频处理企业资料。
3. PostgreSQL / Redis 启停逻辑不够可复制，导致本机运行态波动。

## 4. 候选方案比较

### 方案 A：继续保留现状，只补文档说明

核心思路：

- 不改实现，只在记录里告诉后续使用者“这些地方要手工注意”。

优点：

- 改动最小。
- 不影响现有功能。

缺点：

- 不能真正消除风险。
- `prompt/confirm` 和远程字体依赖仍然存在。
- 本机运行恢复仍靠人记忆。

放弃原因：

- 这类问题已经不是靠文档提醒就能接受的程度，必须做代码级修补。

### 方案 B：针对最影响商用感受的非接口问题做集中收口

核心思路：

- 去掉远程字体依赖。
- 把企业资料页的审核与删除改成正式弹层。
- 把 PostgreSQL / Redis 补成可重复启停脚本，并恢复本机运行态。

优点：

- 改动集中、收益高。
- 不改变业务主结构。
- 对后续 AI 联调和自动链运行有直接帮助。

缺点：

- 需要同时修改前端和本机脚本。
- 需要实际启动本机服务做验证。

最终选择原因：

- 这是当前收益最高、同时又不需要重构业务架构的方案。

## 5. 最终决策

本轮最终采用方案 B，并做了以下关键决策：

1. 去掉 `next/font/google`，改为本地字体栈，避免构建依赖外网。
2. 企业资料页不再使用浏览器原生对话框，统一切到页内审核弹层与删除弹层。
3. PostgreSQL 脚本优先支持仓库内运行时和数据目录自动探测。
4. Redis 补成独立本机脚本，统一走仓库内 Memurai 运行时。
5. 本轮不继续扩新业务能力，优先把已有能力做到更稳定、更像正式后台。

## 7. 本轮优点

相比之前，本轮的改进主要有：

1. 构建不再依赖 Google Fonts 网络资源，离线和受限网络环境更稳定。
2. 企业资料审核页更像正式审核台，而不是开发临时工具。
3. 删除操作不再是原生确认框，风险提示更清晰。
4. 本机 PostgreSQL / Redis 的恢复方式更统一，便于后续继续迭代和演示。
5. 这轮改动没有破坏已有 AI Provider 层和企业资料审核链。

改善的质量属性：

- 稳定性
- 可维护性
- 可运行性
- 使用体验

## 8. 本轮缺点与代价

这一轮仍然有真实 trade-off：

1. 企业资料页虽然已经不再使用 `prompt/confirm`，但目前弹层仍是自实现，不是统一通用 Dialog 体系。
2. PostgreSQL 启动脚本在当前 Codex 沙箱里仍需要脱离沙箱才能真正常驻，这属于工具环境约束，不是业务代码缺陷。
3. Redis 现在使用仓库内 Memurai 运行时，适合本机开发和演示，但还不是正式生产部署方案。
4. `companies/page-client.tsx` 这轮为了快速收口问题，做了整文件重写，虽然功能更干净，但后续仍可以继续拆分局部子组件。

3. PostgreSQL / Redis 本机运行态已经恢复，队列和 dev 服务可以正常工作。

仍需注意：

- PostgreSQL 启动脚本在 Codex 当前沙箱里验证时需要提权启动后台进程。

## 10. 本轮结论

本轮最重要的结论是：

- 到这个阶段，影响“商用感”的主要问题已经不再是有没有功能，而是构建稳定性、审核交互和运行底盘是否可靠。

被证实的假设：

1. 去掉 Google Fonts 依赖后，构建稳定性明显更好。
2. 企业资料审核页只要切掉原生对话框，整体观感和可操作性会立刻提升。
3. Redis 与 PostgreSQL 的本机恢复一旦脚本化，后续验证成本会明显下降。

以后应默认继承的经验：

- 高优先级后台页不要继续使用浏览器原生 `prompt/confirm`。
- 构建依赖应优先选择本地可控资源，而不是外网资产。
- 本机运行环境恢复脚本本身也是正式资产，不应只靠口头说明。

## 11. 下一轮建议

下一轮最值得继续的 3 件事：

1. 把其他仍然依赖 `prompt/confirm` 的页面一并替换成正式弹层。
2. 给 Redis / PostgreSQL 运行脚本补更明确的状态检测与错误提示。
3. 继续把 AI 调用日志接入学习反馈中心，做成功率/失败率统计面板。
