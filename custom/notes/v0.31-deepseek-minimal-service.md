# 版本记录：v0.31-deepseek-minimal-service

## 1. 本轮目标

本轮目标是为“整木网 AI 编辑系统”补一个 **DeepSeek API 最小打通版**，但它不是主系统里的新模块，而是一个独立、最小、可复制运行的 Node.js 服务。

这一轮值得优先处理，原因有三点：

1. 当前主仓库已经较复杂，继续在主系统里做最小联通验证，会把“验证 API 可用”与“现有业务逻辑问题”混在一起。
2. 用户明确要求的是“先把 DeepSeek API 接口成功打通”，而不是继续做完整系统。
3. 需要一个后续任何人都能快速复制、启动、用 curl 验证的最小样板。

本轮属于：

- 功能补充
- 实验基线建立
- 可复用样板补齐

## 2. 背景与上下文

本轮承接：

- `custom/notes/v0.30-commercial-readiness-ui-runtime-hardening.md`
- `custom/notes/experiments/exp-20260328-deepseek-api-smoke-test.md`
- `custom/notes/experiments/exp-20260328-deepseek-copy-generation-test.md`

修改前的状态是：

1. 主仓库已经能用 `openai` SDK 直连 DeepSeek 做临时脚本测试。
2. DeepSeek 联通性已经被实验记录验证过，但缺少一个真正独立、最小、可运行的服务目录。
3. 用户这轮要求明确收束为：
   - Express
   - dotenv
   - cors
   - openai SDK
   - `/api/health`
   - `/api/chat`
   - 请求日志
   - 超时控制
   - 统一错误处理中间件
   - `services/deepseek.js`

已知约束：

- 不做数据库
- 不做登录
- 不做前端
- 不做任务系统
- 不做复杂架构
- 不把 API Key 暴露到前端

## 3. 问题分析

表面问题是“缺一个最小服务”，但真正的核心矛盾是：

- 现有仓库太大，若不把 DeepSeek 最小联通样板独立出来，后续任何测试都会被主系统上下文干扰。

当前最需要避免的问题：

1. 把最小验证继续塞进主系统，导致用户无法快速复制运行。
2. 为了“更完整”引入过多结构，反而违背最小化目标。
3. 忽略最基础但实际非常关键的非功能需求：
   - 请求日志
   - 超时控制
   - 统一错误处理

## 4. 候选方案比较

### 方案 A：直接在主 Next.js 项目里新加 API Route

核心思路：

- 把 `/api/chat` 和 `/api/health` 加到当前主系统里。

优点：

- 复用现有依赖。
- 代码量更少。

缺点：

- 不再是一个“最小独立服务”。
- 会和当前 3000 端口的 Next.js 服务耦合。
- 很难作为独立样板直接复制给别人验证。

放弃原因：

- 不符合用户“最小可运行 Node.js 服务”的要求。

### 方案 B：在 `custom/` 下新建独立最小服务目录

核心思路：

- 单独创建 `custom/deepseek-minimal/`
- 自带 `package.json / server.js / services/deepseek.js / .env.example / README.md`

优点：

- 边界清晰，不污染主系统。
- 便于复制、启动、测试和后续独立扩展。
- 保持最小架构同时满足新增的日志、超时和错误处理中间件要求。

缺点：

- 需要额外一份独立 `package.json`
- 与主仓库已有依赖存在一定重复

最终选择原因：

- 这是最符合“最小独立验证服务”目标的方案。

## 5. 最终决策

本轮最终采用方案 B，并做了这些关键决策：

1. 服务目录放到 `custom/deepseek-minimal/`，不入侵现有业务目录。
2. 使用 CommonJS，减少 Node 版本和配置摩擦。
3. 端口默认设为 `3000`，直接对齐用户给的 curl 示例。
4. 同时在 README 中明确说明：若当前主项目占用 `3000`，可改 `.env` 中的 `PORT`。
5. 超时控制保持最小实现，用 `Promise.race`，不额外引入复杂取消机制。
6. 服务层只保留一个 `services/deepseek.js`，不继续抽象 provider、模板或中间层。

## 6. 具体实现

本轮新增文件：

- `custom/deepseek-minimal/package.json`
- `custom/deepseek-minimal/server.js`
- `custom/deepseek-minimal/services/deepseek.js`
- `custom/deepseek-minimal/.env.example`
- `custom/deepseek-minimal/README.md`

本轮还修改：

- `AGENTS.md`

各文件作用：

- `package.json`：提供最小依赖和启动脚本。
- `server.js`：承载 Express 服务、请求日志、超时控制、接口定义和统一错误处理中间件。
- `services/deepseek.js`：负责创建 DeepSeek 客户端并封装聊天调用。
- `.env.example`：给出最小运行所需环境变量。
- `README.md`：写清安装、配置、启动、curl 测试和常见报错排查。
- `AGENTS.md`：同步最新正式记录路径。

## 7. 本轮优点

相比之前，本轮新增的最主要价值是：

1. 仓库里第一次有了一个完全独立的 DeepSeek 最小联通服务样板。
2. 用户现在不需要理解整个主系统，也能快速验证 DeepSeek API 是否打通。
3. 请求日志、超时控制和统一错误处理已经纳入最小样板，不是只有“能调通”的临时脚本。
4. 服务层封装保持最小，但已经为后续继续扩到更多接口预留了一个自然位置。

改善的质量属性：

- 可复制性
- 可验证性
- 可维护性
- 可演示性

## 8. 本轮缺点与代价

这轮也有明确 trade-off：

1. 这是独立最小服务，不会自动接入现有主系统的 AI Provider 层。
2. 依赖与主仓库存在少量重复，这是为了独立运行换来的代价。
3. 超时控制使用的是最小 Promise 级别超时，不是更重的底层请求取消方案。
4. 端口默认使用 `3000`，如果与现有 Next.js 服务同时运行，需要用户手动调整 `PORT`。

## 9. 验证与结果

本轮可执行的最小验证包括：

- 使用 `node --check` 做语法检查
- 检查文件结构是否完整
- 校验 README、环境变量模板和 curl 示例是否齐全

由于当前目录是新增最小服务样板，还未执行 `npm install`，因此本轮没有在该目录下直接启动 Express 进程。

这意味着：

- 语法和结构可以验证
- 真正联网调用 DeepSeek 仍需用户在该目录执行一次 `npm install` 后再启动

## 10. 本轮结论

本轮最重要的结论是：

- DeepSeek API 的“最小打通版本”最适合以独立服务的方式存在，而不是继续嵌进已经复杂的主系统。

被确认的经验：

1. 最小联通样板应当独立存放，避免被主系统上下文拖累。
2. 即使是最小服务，也至少要带日志、超时和统一错误处理。
3. 对当前仓库而言，`custom/` 是放这类独立样板的合适位置。

## 11. 下一轮建议

下一步最值得继续的 3 件事：

1. 在 `custom/deepseek-minimal/` 目录实际执行 `npm install && npm run dev`，完成真实本地联通验证。
2. 基于这个最小样板，继续补一个 `/api/chat/stream` 流式版本。
3. 如果后续要回接主系统，再把其中成熟的错误处理和超时控制策略抽回统一 AI Provider 层。
